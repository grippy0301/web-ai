# Web制作サービス Phase 1 ロードマップ_JP

> Notion: [元ページ](https://www.notion.so/Web-Phase-1-_JP-367089a9cb07804e9256c8d6f7b0ecb1)

> フェーズ名: Phase 1 リード発掘・スクリーニング

---

## **1. フェーズ概要**
Phase 1 の目的は、**Phase 2 がすぐ営業と試作提案に進める見込み店舗を、毎週安定して供給すること**である。ここで重要なのは、候補を大量に集めることではなく、ソロ運営でも無理なく回せる件数に絞り、後工程で使える精度まで整えることにある。
このフェーズの成果物は次の3つに集約する。
- A/B グレード付きのリード一覧
- A グレード優先リードの handoff 用 JSON
- 次週に改善すべき検索条件と除外ルールのメモ
運用方針は「広く取る」ではなく「狭く正確に回して学習する」。開始直後は対象地域と業種を絞り、4週連続で安定運用できてから拡張する。
**判断ルール**
- 迷ったリードは A に上げず B に留める
- 自動判定だけで営業対象にしない。A は必ず人が最終確認する
- Phase 1 の役割は「優先順位付け」まで。営業文面や提案作成まで持ち込まない

## **3. 前提条件**
このフェーズを始める前に、少なくとも以下が揃っている必要がある。

| （Notion表 — 元ページで確認） |
| --- |

**推奨の開始条件**
- 地域は商圏が読みやすい都市部 1〜2 エリアから始める
- 業種は飲食、美容、整体/整骨院など、需要判定が比較的しやすいものから始める
- 初月は 1 回の実行を `最大 6 クエリ` までに抑える
**開始前の判断ルール**
- API キー制限と予算アラートがない場合は実行しない
- Phase 2 の必要項目が曖昧なら先に handoff フォーマットを固定する
- 対象地域を増やす前に、既存地域で 2 週連続の運用実績を作る

## **4. スコープと非ゴール**
### **スコープ**
- Google Maps の公式 API から候補店舗を取得する
- 基本条件で候補を絞り込む
- Web サイトの有無と品質をスコアリングする
- 営業優先度に応じて A/B/C に分ける
- A グレードを人が確認し、Phase 2 へ渡せる形に整える
### **非ゴール**
- すべての地域・業種を網羅すること
- 完全自動で営業対象を確定すること
- 店舗オーナーの個人情報を追加収集すること
- Google Maps の HTML スクレイピングを行うこと
- 提案文、試作サイト、営業送信までをこのフェーズで完結させること
**線引きのルール**
- 既存サイトの良し悪しで悩む案件は B に留め、Phase 2 に渡さない
- 営業の当たり外れを Phase 1 の責任にしすぎない。Phase 1 の責任範囲は「見込みのある候補を揃える」まで
- 1件ずつ深掘りしすぎない。A/B/C 分類に必要な情報だけを集める

## **5. 実行ワークフロー**
Phase 1 は、毎回同じ順序で回せることが重要である。以下を標準フローとする。
### **5-1. 対象設定**
1 回の実行では、地域と業種の掛け合わせを小さく保つ。
- 初月の標準: `2地域 × 2〜3業種`
- 1実行あたりの検索対象は `最大 6 クエリ`
- 1クエリで候補が多すぎる場合は半径を狭める
**判断ルール**
- 1クエリで 60 件近く取得される場合は、次回から半径を 20〜30% 縮小する
- 1クエリで 10 件未満しか出ない場合は、業種語を変えるか Text Search に切り替える
### **5-2. 候補取得**
Google Maps Places API の Nearby Search または Text Search で候補を取得する。`website` は含まれないため、候補取得の段階では「可能性のある店舗一覧」を作ることが目的になる。
**判断ルール**
- `next_page_token` がある場合のみ追加取得する
- 追加ページの取得間隔は 2.5 秒以上空ける
- `business_status` が `OPERATIONAL` 以外はこの時点で除外する
### **5-3. 基本フィルタ**
候補取得後、以下を満たすものだけを詳細確認に進める。
- レビュー数 `5件以上`
- 評価 `3.8以上`
- `business_status = OPERATIONAL`
**判断ルール**
- レビュー 4 件以下は原則除外。知名度が弱く営業優先度が低い
- 評価 3.7 以下は除外。受注後に実績化しにくい
### **5-4. 詳細取得**
Place Details で `website`、`phone`、`rating`、`user_ratings_total` を取得する。
**判断ルール**
- `website = null` は即 A 判定の候補にする
- 電話番号が欠落していても A/B 判定は可能。ただし Phase 2 渡し前に欠落を明示する
- `place_id` は必ず保持し、重複判定キーとして扱う
### **5-5. サイト品質チェック**
既存サイトがある店舗に対してのみ、次の観点を確認する。
- PageSpeed モバイルスコア
- SSL 対応の有無
- viewport メタタグの有無
- Wayback Machine による更新停滞の推定
**標準スコアリング**

| （Notion表 — 元ページで確認） |
| --- |

**グレード判定**
- `3点以上`: A
- `1〜2点`: B
- `0点`: 除外
**例外時のルール**
- PageSpeed 失敗だけでは加点しない。`要目視` として記録する
- HTML 取得失敗だけでは A にしない。既存サイトあり案件として B に寄せる
- URL 不正は A 候補にできるが、Phase 2 へ渡す前に人が確認する
### **5-6. 人による最終確認**
A 判定はすべて目視で確認する。確認する項目は以下の通り。
- 廃業、移転、臨時休業が明らかでないか
- 店舗種別が営業対象から大きく外れていないか
- 既存サイトの改善余地が実際にあるか
- 電話番号や住所が極端に欠損していないか
**判断ルール**
- 10秒見て判断できない案件は B に下げる
- 「古いが十分使えるサイト」は B に留める
- A に残すのは Phase 2 がその週に着手できる件数まで
### **5-7. 出力と記録**
最終的に以下を出力する。
- 全件 CSV
- A グレード優先 JSON
- 今週の除外理由メモ
**判断ルール**
- A グレードは 1 回の handoff で `5〜15件` を目安にする
- 15件を超える場合はスコア順、業種集中度、営業しやすさで絞る
- 同一店舗の再抽出は `place_id` ベースで統合する

## **6. 週次運用カデンス**
ソロ運営では、毎日触るより「週1の定例運用 + 週中の軽い見直し」が安定する。

| （Notion表 — 元ページで確認） |
| --- |

**週次の標準リズム**
- 月曜: 取る、絞る、渡す
- 水曜: 直す
- 金曜: 判断材料を残す
**運用ルール**
- 月曜の処理が 45 分を超えたら翌週の対象範囲を減らす
- 水曜の見直しで除外理由が毎週同じなら、ルール化して自動除外に寄せる
- 金曜にコスト確認を飛ばした週は、翌週の検索範囲を増やさない

### **コンプライアンス上の明確なルール**

## **9. Phase 2 への引き継ぎ**
Phase 1 の完了は、A グレードの候補を「営業に使える単位」にして渡した時点とする。CSV を作っただけでは完了ではない。
### **handoff 対象**
- A グレードのみ
- その週に Phase 2 が着手できる件数のみ
- 人の最終確認を通過したもののみ
### **handoff に最低限必要な項目**

| （Notion表 — 元ページで確認） |
| --- |

### **handoff ルール**
- `website = null` は「新規立ち上げ提案」軸で渡す
- `website` ありは「改善提案」軸で渡す
- `reasons` は営業に使うため、1件につき 1〜3 個に絞る
- `pagespeed_score = null` の案件は、営業根拠を別の理由で補強できる場合のみ渡す
- 電話番号が欠けていてもよいが、欠けている事実を明記する
### **Phase 2 に渡す際の一言メモの標準**
- 何が問題か
- なぜ今アプローチ価値があるか
- どの角度で試作提案すると刺さりそうか

---

## **10. 30/60/90日展開**

| （Notion表 — 元ページで確認） |
| --- |

### **0〜30日**
- スケールしない
- 誤判定を減らす
- どの業種が当たりやすいかを見る
### **31〜60日**
- B に落とした案件の傾向を見て、A 条件を明文化する
- 手動で毎回やっている除外をルール化する
- Phase 2 から「使える A」と「弱い A」の差分を回収する
### **61〜90日**
- 成果の出る地域・業種に集中して横展開する
- 週次の定例運用を崩さずに、検索対象だけを広げる
- この段階でも手動時間が増えるなら、拡張より絞り込みを優先する
