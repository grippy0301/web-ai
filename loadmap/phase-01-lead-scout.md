## **1. フェーズ概要**

Phase 1 の目的は、**Phase 2 がすぐ営業と試作提案に進める見込み店舗を、毎週安定して供給すること**である。ここで重要なのは、候補を大量に集めることではなく、ソロ運営でも無理なく回せる件数に絞り、後工程で使える精度まで整えることにある。

このフェーズの成果物は次の3つに集約する。

- A/B グレード付きのリード一覧
- A グレード優先リードの handoff 用 JSON
- 次週に改善すべき検索条件と除外ルールのメモ

運用方針は「広く取る」ではなく「狭く正確に回して学習する」。開始直後は対象地域と業種を絞り、4週連続で安定運用できてから拡張する。

**判断ルール**

- 迷ったリードは A に上げず B に留める
- 自動判定だけで営業対象にしない。A は必ず人が最終確認する
- Phase 1 の役割は「優先順位付け」まで。営業文面や提案作成まで持ち込まない

## **2. 成功指標**

| 指標 | 目標 | 下限ライン | 未達時の対応 |
| --- | --- | --- | --- |
| 月間抽出リード数 | 30〜50件 | 20件 | 対象地域を1つ追加する前に検索語と業種を見直す |
| 月間 A グレード数 | 10〜30件 | 8件 | スコア閾値は維持し、検索対象の質を改善する |
| 週次の手動作業時間 | 30分以内 | 45分以内 | 45分を超えたら対象件数を削る |
| A リードの Phase 2 受入率 | 80%以上 | 70%以上 | 除外ルールを追加し、A判定の条件を厳格化する |
| 重複率 | 10%未満 | 15%未満 | `place_id` で重複除去を強化する |
| 月次 API コスト | 実質 $0〜15 | $30以内 | $20で警戒、$30で拡張停止 |

**運用上の見方**

- 月間抽出数よりも A グレードの受入率を優先する
- A グレードが多くても Phase 2 で弾かれるなら、Phase 1 は失敗とみなす
- 週次作業が重くなった時点で、自動化ではなく探索範囲の縮小を先に行う

## **3. 前提条件**

このフェーズを始める前に、少なくとも以下が揃っている必要がある。

| 項目 | 必須内容 |
| --- | --- |
| API 利用準備 | Google Maps Places API、Place Details、PageSpeed Insights API を使える状態 |
| 課金管理 | Google Cloud の請求設定と月次アラート設定 |
| データ保管先 | CSV 出力先と、確認用の Notion または同等の台帳 |
| 対象定義 | 最初の対象地域 1〜2件、対象業種 2〜3件 |
| 受け渡し仕様 | Phase 2 が必要とする最低項目が確定していること |
| 保持ルール | リードデータを 30 日以内で再取得または削除する運用ルール |

**推奨の開始条件**

- 地域は商圏が読みやすい都市部 1〜2 エリアから始める
- 業種は飲食、美容、整体/整骨院など、需要判定が比較的しやすいものから始める
- 初月は 1 回の実行を `最大 6 クエリ` までに抑える

**開始前の判断ルール**

- API キー制限と予算アラートがない場合は実行しない
- Phase 2 の必要項目が曖昧なら先に handoff フォーマットを固定する
- 対象地域を増やす前に、既存地域で 2 週連続の運用実績を作る

## **4. スコープと非ゴール**

### **スコープ**

- Google Maps の公式 API から候補店舗を取得する
- 基本条件で候補を絞り込む
- Web サイトの有無と品質をスコアリングする
- 営業優先度に応じて A/B/C に分ける
- A グレードを人が確認し、Phase 2 へ渡せる形に整える

### **非ゴール**

- すべての地域・業種を網羅すること
- 完全自動で営業対象を確定すること
- 店舗オーナーの個人情報を追加収集すること
- Google Maps の HTML スクレイピングを行うこと
- 提案文、試作サイト、営業送信までをこのフェーズで完結させること

**線引きのルール**

- 既存サイトの良し悪しで悩む案件は B に留め、Phase 2 に渡さない
- 営業の当たり外れを Phase 1 の責任にしすぎない。Phase 1 の責任範囲は「見込みのある候補を揃える」まで
- 1件ずつ深掘りしすぎない。A/B/C 分類に必要な情報だけを集める

## **5. 実行ワークフロー**

Phase 1 は、毎回同じ順序で回せることが重要である。以下を標準フローとする。

### **5-1. 対象設定**

1 回の実行では、地域と業種の掛け合わせを小さく保つ。

- 初月の標準: `2地域 × 2〜3業種`
- 1実行あたりの検索対象は `最大 6 クエリ`
- 1クエリで候補が多すぎる場合は半径を狭める

**判断ルール**

- 1クエリで 60 件近く取得される場合は、次回から半径を 20〜30% 縮小する
- 1クエリで 10 件未満しか出ない場合は、業種語を変えるか Text Search に切り替える

### **5-2. 候補取得**

Google Maps Places API の Nearby Search または Text Search で候補を取得する。`website` は含まれないため、候補取得の段階では「可能性のある店舗一覧」を作ることが目的になる。

**判断ルール**

- `next_page_token` がある場合のみ追加取得する
- 追加ページの取得間隔は 2.5 秒以上空ける
- `business_status` が `OPERATIONAL` 以外はこの時点で除外する

### **5-3. 基本フィルタ**

候補取得後、以下を満たすものだけを詳細確認に進める。

- レビュー数 `5件以上`
- 評価 `3.8以上`
- `business_status = OPERATIONAL`

**判断ルール**

- レビュー 4 件以下は原則除外。知名度が弱く営業優先度が低い
- 評価 3.7 以下は除外。受注後に実績化しにくい

### **5-4. 詳細取得**

Place Details で `website`、`phone`、`rating`、`user_ratings_total` を取得する。

**判断ルール**

- `website = null` は即 A 判定の候補にする
- 電話番号が欠落していても A/B 判定は可能。ただし Phase 2 渡し前に欠落を明示する
- `place_id` は必ず保持し、重複判定キーとして扱う

### **5-5. サイト品質チェック**

既存サイトがある店舗に対してのみ、次の観点を確認する。

- PageSpeed モバイルスコア
- SSL 対応の有無
- viewport メタタグの有無
- Wayback Machine による更新停滞の推定

**標準スコアリング**

| 条件 | 加点 |
| --- | --- |
| Web サイトなし | +3 |
| PageSpeed モバイル 49 以下 | +2 |
| PageSpeed モバイル 50〜69 | +1 |
| SSL 非対応または URL 不正 | +2 |
| viewport 未設定 | +1 |
| 3年以上更新なしと推定 | +1 |

**グレード判定**

- `3点以上`: A
- `1〜2点`: B
- `0点`: 除外

**例外時のルール**

- PageSpeed 失敗だけでは加点しない。`要目視` として記録する
- HTML 取得失敗だけでは A にしない。既存サイトあり案件として B に寄せる
- URL 不正は A 候補にできるが、Phase 2 へ渡す前に人が確認する

### **5-6. 人による最終確認**

A 判定はすべて目視で確認する。確認する項目は以下の通り。

- 廃業、移転、臨時休業が明らかでないか
- 店舗種別が営業対象から大きく外れていないか
- 既存サイトの改善余地が実際にあるか
- 電話番号や住所が極端に欠損していないか

**判断ルール**

- 10秒見て判断できない案件は B に下げる
- 「古いが十分使えるサイト」は B に留める
- A に残すのは Phase 2 がその週に着手できる件数まで

### **5-7. 出力と記録**

最終的に以下を出力する。

- 全件 CSV
- A グレード優先 JSON
- 今週の除外理由メモ

**判断ルール**

- A グレードは 1 回の handoff で `5〜15件` を目安にする
- 15件を超える場合はスコア順、業種集中度、営業しやすさで絞る
- 同一店舗の再抽出は `place_id` ベースで統合する

## **6. 週次運用カデンス**

ソロ運営では、毎日触るより「週1の定例運用 + 週中の軽い見直し」が安定する。

| 曜日 | 作業 | 所要時間 | 目的 |
| --- | --- | --- | --- |
| 月曜朝 | スクリプト実行と一次出力 | 15〜20分 | 候補の取得と初期分類 |
| 月曜朝 | A 判定の目視確認 | 15〜20分 | 誤判定と不要案件の除去 |
| 月曜午前 | Phase 2 向け handoff 作成 | 10分 | 営業着手可能な形に整える |
| 水曜 | 除外理由と不足データの見直し | 10分 | 次回の検索条件改善 |
| 金曜 | KPI とコスト確認 | 10分 | 運用負荷と採算の維持 |

**週次の標準リズム**

- 月曜: 取る、絞る、渡す
- 水曜: 直す
- 金曜: 判断材料を残す

**運用ルール**

- 月曜の処理が 45 分を超えたら翌週の対象範囲を減らす
- 水曜の見直しで除外理由が毎週同じなら、ルール化して自動除外に寄せる
- 金曜にコスト確認を飛ばした週は、翌週の検索範囲を増やさない

## **7. ツール/API スタックとコストガードレール**

### **使用スタック**

| ツール/API | 用途 | 運用上の注意 |
| --- | --- | --- |
| Google Maps Places API | 候補店舗の検索 | 検索範囲を広げすぎない |
| Place Details | `website`、電話番号、評価取得 | 必要フィールドのみに絞る |
| PageSpeed Insights API | モバイル性能の確認 | タイムアウトを長めに取る |
| Wayback Machine API | 更新停滞の推定 | 連続呼び出しを抑える |
| Node.js + axios | 実行基盤 | 失敗理由をログに残す |
| CSV/Notion | 目視確認と handoff 管理 | 30日超のデータを残しすぎない |

### **コスト目安**

元資料の前提では、週1運用で月次コストは概算 `$14.54`、Google Maps の無料クレジット内に収まる想定である。実運用では「無料枠内に収める」よりも、「どこで止めるか」を先に決めておく。

### **コストガードレール**

| 項目 | 目標 | 上限 | 対応 |
| --- | --- | --- | --- |
| 月次 API コスト | $0〜15 | $30 | $20 到達で警戒、$30 で拡張停止 |
| 1回あたり検索クエリ数 | 4〜6 | 9 | 9 を超える拡張は 2 週連続安定後のみ |
| 1回あたり Place Details 件数 | 50〜120 | 180 | 180 を超えそうなら基本フィルタを先に厳格化 |
| 1回あたり PageSpeed 実行件数 | 30〜80 | 100 | 100 超は B 判定以下をサンプリングに切り替える |

**運用ルール**

- 初月は無料枠に余裕があっても規模を広げない
- Place Details の `fields` は必要最小限に固定する
- ターゲット拡張は「A 受入率 80%以上」が 2 週続いたときだけ行う

## **8. リスク登録簿とコンプライアンスガードレール**

| リスク | 兆候 | 影響 | 予防策 | 対応策 |
| --- | --- | --- | --- | --- |
| Google 利用規約違反 | データを長期保存している | API 利用停止リスク | 30日以内の再取得または削除を徹底 | 古いリードを週次で棚卸しする |
| Google Maps の非公式取得 | HTML スクレイピングをしたくなる | 重大 | 公式 API 以外を使わない | 要件から外す |
| レート制限超過 | エラーが連続する | 実行失敗 | リクエスト間隔を固定 | 実行を止めて翌日再試行 |
| 重複リード | 同一店舗が複数回出る | Phase 2 の無駄工数 | `place_id` で統合 | 重複率を週次確認する |
| 廃業・実態不一致 | Maps は営業中だが実態が違う | 誤営業 | A は必ず目視確認 | `rejected` 理由を記録し除外ルールに反映 |
| サイト評価の誤判定 | JS サイトで viewport 判定が外れる | A/B のノイズ増加 | 例外は `要目視` に逃がす | スコア加点を抑える |
| 個人情報の扱い | 電話番号を雑に共有する | 管理リスク | 社外共有しない | 共有先を Phase 2 のみへ限定 |
| ソロ運営の負荷増大 | 週次作業が長引く | 継続不能 | 件数上限を先に決める | 範囲縮小を優先する |

### **コンプライアンス上の明確なルール**

- Google Maps のデータ取得は必ず公式 API を使う
- 保存したリードデータは 30 日を超えて残さない
- Google 由来データを公開リポジトリに置かない
- 店舗の公開情報だけを扱い、個人の私的情報を追加しない
- 営業対象にする前に、人が最低限の妥当性確認を行う

## **9. Phase 2 への引き継ぎ**

Phase 1 の完了は、A グレードの候補を「営業に使える単位」にして渡した時点とする。CSV を作っただけでは完了ではない。

### **handoff 対象**

- A グレードのみ
- その週に Phase 2 が着手できる件数のみ
- 人の最終確認を通過したもののみ

### **handoff に最低限必要な項目**

| 項目 | 必須 | 用途 |
| --- | --- | --- |
| `lead_id` | 必須 | 追跡用 |
| `name` | 必須 | 提案対象の識別 |
| `address` | 必須 | 商圏確認 |
| `phone` | 推奨 | 連絡手段 |
| `website` | 必須 | 新規制作か改善提案かの判断 |
| `rating` | 必須 | 営業優先度の補足 |
| `review_count` | 必須 | 需要の強さを見る |
| `pagespeed_score` | 推奨 | 改善提案の根拠 |
| `reasons` | 必須 | 営業トークの切り口 |
| `place_id` | 必須 | 店舗ページ参照用 |
| `extracted_at` | 必須 | 30日ルール管理用 |

### **handoff ルール**

- `website = null` は「新規立ち上げ提案」軸で渡す
- `website` ありは「改善提案」軸で渡す
- `reasons` は営業に使うため、1件につき 1〜3 個に絞る
- `pagespeed_score = null` の案件は、営業根拠を別の理由で補強できる場合のみ渡す
- 電話番号が欠けていてもよいが、欠けている事実を明記する

### **Phase 2 に渡す際の一言メモの標準**

- 何が問題か
- なぜ今アプローチ価値があるか
- どの角度で試作提案すると刺さりそうか

## **10. 30/60/90日展開**

| 期間 | 目的 | やること | 進捗判定 |
| --- | --- | --- | --- |
| 0〜30日 | 再現性の確立 | 2地域×2〜3業種で週1運用、A/B/C判定のブレを減らす | 4週連続で手動 45 分以内 |
| 31〜60日 | 精度改善 | 除外ルール追加、重複削減、A 受入率改善、対象業種の当たり外れを比較 | A 受入率 80%以上が 2 週連続 |
| 61〜90日 | 拡張準備 | 地域または業種を 1 つ増やす、handoff の定型化、Phase 2 からの逆フィードバック反映 | 月間 A グレード 15件以上を安定供給 |

### **0〜30日**

- スケールしない
- 誤判定を減らす
- どの業種が当たりやすいかを見る

### **31〜60日**

- B に落とした案件の傾向を見て、A 条件を明文化する
- 手動で毎回やっている除外をルール化する
- Phase 2 から「使える A」と「弱い A」の差分を回収する

### **61〜90日**

- 成果の出る地域・業種に集中して横展開する
- 週次の定例運用を崩さずに、検索対象だけを広げる
- この段階でも手動時間が増えるなら、拡張より絞り込みを優先する
