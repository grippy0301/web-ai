> 対象: 一人運営のWeb制作事業者
> 
> 
> フェーズ位置: Phase 2の受注見込み案件を、契約締結・入金確定・要件凍結まで進めてPhase 4へ引き渡すための実行フェーズ
> 
> 参照元: `new/web_creation_service_roadmap_JP.md` の Phase 3、および `new/agents/03_contract_advisor.md`

---

## **1. フェーズ概要**

Phase 3の目的は、案件を「受注したつもり」の状態から「着手してよい状態」へ変えることにある。

一人運営では、営業時の勢いで開発に入り、後から未払い・仕様追加・認識ズレが発生しやすい。このフェーズではそれを防ぐため、契約、入金、ヒアリング、要件定義、変更ルール説明を短期間で完了させる。

このフェーズの標準所要期間は **3〜7営業日**。

出口は「契約が成立していること」ではなく、**Phase 4が迷わず着手できること** と定義する。

### **Phase 3の出口成果物**

- 電子署名済み契約
- 入金確認済みの着手金
- 回答済みヒアリングシート
- 承認済み要件定義サマリー
- 変更依頼ルールの事前合意
- Phase 4向け引き渡しパック

---

## **2. 成功指標**

| 指標 | 目標値 | 判定ルール |
| --- | --- | --- |
| 契約パック送付までの時間 | 1営業日以内 | 発注意思を受けた翌営業日中に送付 |
| 電子署名完了率 | 80%以上 | 契約送付案件のうち署名完了した割合 |
| 着手金回収率 | 100% | 入金確認前に着手しない運用を守る |
| ヒアリング回収完了日数 | 3営業日以内 | フォーム送付から必須回答回収まで |
| 高優先度の曖昧要件残数 | 0件 | Phase 4引き渡し時点でゼロ |
| Phase 3総リードタイム | 7営業日以内 | 発注意思確認から要件凍結まで |
| Phase 4開始後の重大仕様変更率 | 20%未満 | ページ追加、機能追加、方向性全面変更の発生率 |

## **3. 前提条件**

Phase 3に入る前に、最低限以下が揃っていることを確認する。

- 顧客から「進めたい」「見積を前提に具体化したい」という発注意思がある
- 予定プランがライト / スタンダード / プレミアムのいずれかに仮置きされている
- 決定権者が誰か明確になっている
- 主連絡先が `メール` か `チャット` のどちらかに固定されている
- 公開希望時期が大まかでも把握できている
- 試作サイトまたは提案内容を顧客が一度確認している
- 反社、支払い不安、過度な値引き要求などの赤信号が出ていない

### **進行停止にする条件**

- 予算感が極端に合わない
- 決定権者が不在のまま話が進んでいる
- 公開希望日が現実的でないのに調整余地がない
- 契約や支払いの話を避け続ける
- LINE/口頭のみで進めたいなど、証跡を残す運用に非協力的

---

## **4. スコープと非目標**

### **このフェーズでやること**

- 受注判断の最終確認
- 見積、契約、請求の送付
- 着手金の回収
- ヒアリングシート回収と補足ヒアリング
- AIを使った要件整理
- 曖昧さ、矛盾、予算超過要素の解消
- 変更依頼ルール、承認ルール、停止ルールの共有
- Phase 4へ渡すための要件凍結

### **このフェーズでやらないこと**

- コーディング着手
- デザイン作業の本制作
- SEO順位や集客成果の保証
- 写真撮影や原稿制作のフル実施
- 契約未了状態でのスケジュール確約
- 入金前の優先枠確保

## **5. 実行フロー**

Phase 3は、契約書を送ること自体が目的ではない。

**「契約成立 → 入金確定 → 要件凍結」までを1本の流れとして運用する。**

### **標準タイムライン**

| 日程目安 | ステップ | 到達条件 |
| --- | --- | --- |
| Day 0 | 受注意思確認、Go/No-Go判定 | 進める / 断る の判断がつく |
| Day 1 | 見積・契約・請求・ヒアリング案内送付 | 契約パック送付完了 |
| Day 1〜3 | 電子署名、着手金回収 | 着手条件が揃う |
| Day 2〜4 | ヒアリングシート回収 | 必須項目の抜け漏れがない |
| Day 3〜5 | 補足ヒアリング実施 | 曖昧項目を口頭で解消 |
| Day 4〜6 | 要件定義サマリー作成 | 顧客確認用の文書ができる |
| Day 5〜7 | 要件承認、変更ルール再確認 | Phase 4引き渡し可 |

### **実行ステップ**

### **Step 1. Go/No-Go判定**

受注の雰囲気だけで進めず、以下を見て判断する。

- 予算と希望機能が大きくズレていないか
- 決定権者が会話に入っているか
- 連絡速度が極端に遅くないか
- 公開希望日が現実的か
- 素材や情報提供に協力的か

**判断ルール:**

- 予算超過が `30%以上` 見込まれる場合は、契約送付前にスコープ圧縮案か上位プランへの切替案を出す
- 決定権者が不在なら、契約前に確認機会を1回入れる
- 着手前から要求が頻繁に増える案件は、変更ルールを先に明文化できない限り進めない

### **Step 2. 契約パック送付**

発注意思確認後、1営業日以内に以下をまとめて送る。

- 見積書
- 契約書
- 支払い案内
- ヒアリングシート
- 進行スケジュールの概要

**判断ルール:**

- 見積の有効期限は `14日` を標準にする
- 契約は原則 `電子契約` を使う
- 「まず作り始めてください」は受けず、署名と入金を優先する

### **Step 3. 契約成立と着手金回収**

このステップが、実務上もっとも重要な防波堤になる。

**判断ルール:**

- `電子署名完了` と `着手金入金確認` の両方が揃うまで着手しない
- チャットでの「お願いします」だけでは着手条件にしない
- 請求期限は発行後 `14日以内`
- 期日超過 `7日` で案件を一時停止、`30日` で解除判断に入る

### **Step 4. ヒアリング回収**

Googleフォームなどで先に情報を集め、必要な補足だけ打ち合わせで確認する。

一人運営では、最初から長時間の打ち合わせをすると非効率になりやすい。

**判断ルール:**

- ヒアリングシートは `先に提出`、打ち合わせは `後から補完`
- 必須回答が揃わない限り要件定義に進まない
- 回答期限は送付から `3営業日`
- 未提出の場合は `1回目リマインド`、さらに `3営業日` 無反応なら案件停止候補にする

### **Step 5. 要件定義サマリー作成**

ヒアリング内容をそのまま鵜呑みにせず、AIで整理した上で制作者視点の判断を加える。

- ページ構成
- 必須機能
- 対象外機能
- デザイン方向性
- 必要素材
- 公開目標日
- 保守有無
- 要確認事項

**判断ルール:**

- 「おしゃれに」「集客できる感じ」などは、そのまま残さず具体表現に変換する
- 予算と機能が矛盾する場合は、必ず `要件を減らす` か `予算を上げる` の二択にする
- 参考サイトが出ない場合は、こちらで `3つの方向性` を提示して選んでもらう

### **Step 6. 要件凍結と承認**

顧客が読みやすい1枚要約を出し、合意点と対象外を明確にする。

**判断ルール:**

- 承認は `メール` か `文書` で残す
- 口頭だけの承認は不可
- 高優先度の曖昧項目が1件でも残るならPhase 4へ渡さない
- 対象外機能と追加費用条件を再度明記する

### **Step 7. Phase 4引き渡し**

契約情報と要件情報をまとめて、開発着手に必要な情報だけを整理して渡す。

**判断ルール:**

- 引き渡し時点で、未解決の blocker は `0件`
- 画像未提出などの保留事項がある場合は、仮素材運用か納期延期かを先に決める
- 仕様未確定のまま「走りながら決める」は採用しない

## **6. 契約・支払いルール**

契約と支払いは、ソロ運営ではキャッシュフローと精神的負荷の両方に直結する。

このフェーズでは、例外運用をできるだけ減らす。

| 項目 | 標準ルール | 運用メモ |
| --- | --- | --- |
| 契約方式 | 電子契約を原則 | 印紙不要、証跡が残る |
| 見積有効期限 | 14日 | 期限切れ後は再見積 |
| 着手条件 | 署名完了 + 着手金入金確認 | どちらか片方では開始しない |
| ライトプラン支払 | 50%着手 / 50%納品 | 小額案件は2分割で管理を簡単にする |
| スタンダード以上支払 | 50%着手 / 30%初稿 / 20%納品 | 進行負荷と回収リスクを両立 |
| 月額保守 | 前払い | 可能ならStripe定期決済 |
| 請求期限 | 請求書発行後14日 | 過ぎたら即フォロー |
| 遅延時対応 | 7日超で停止、30日超で解除判断 | 感情でなくルールで処理する |
| 著作権移転 | 全額入金後 | 未払い時の防御線になる |
| 実績掲載 | 原則可、不可は書面申告 | 後日の認識ズレを防ぐ |

### **実務上の固定ルール**

- 制作費の `全額後払い` は受けない
- 保守費は `翌月分前払い` を基本にする
- 請求と承認の証跡は、`メール` か `電子契約サービス` に集約する
- 修正依頼や仕様変更の受付窓口は `1つ` に固定する
- 料金交渉で値下げする場合は、必ず同時にスコープも削る

## **7. ヒアリングと要件整理フロー**

ヒアリングは、顧客の希望を全部聞く場ではなく、

**Phase 4で迷わない情報に変換する場** として設計する。

### **回収すべき必須情報**

- 店舗名、業種、所在地、連絡先
- サイトの主目的
- 主ターゲット
- 必要ページ
- 必要機能
- 参考サイトまたは好みの方向性
- 写真、ロゴ、原稿の有無
- 公開希望時期
- ドメイン、フォーム、CMS、保守の希望
- 予算帯と決定権者

### **推奨フロー**

### **1. フォーム回答回収**

最初に構造化データを集める。

打ち合わせ前に、何が足りないか見える状態を作る。

### **2. 45〜60分の補足ヒアリング**

フォームで拾えない以下だけを確認する。

- デザインの温度感
- 優先順位
- 競合との差別化
- 使えない表現、避けたい雰囲気
- 納期に関わる事情

### **3. AI整理**

フォーム回答と会話メモをAIに渡し、以下を出す。

- プロジェクト概要
- ページ構成
- 機能要件
- デザイン要件
- 対象外
- 要確認事項

### **4. Ambiguity Check**

以下に該当したら、解消するまで凍結しない。

- 参考サイトなしでデザイン要望が抽象的
- 予算より上の機能が含まれる
- 素材がないのに写真中心デザインを希望
- 公開希望日が短すぎる
- 決定権者の承認がまだない

### **5. 要件凍結**

顧客向けに短くまとめた承認版を送る。

ここで「含まれるもの」と「含まれないもの」を同じ重みで書く。

### **要件凍結の具体ルール**

- 参考URLは `最低1件`、理想は `3件`
- 写真素材が未確定なら、`顧客提供 / 有料素材 / AI生成 / 仮置き` のどれにするか先に選ぶ
- ブログ、予約、EC、多言語は `有無` だけでなく `実装方式` まで決める
- ドメイン未定なら、`新規取得` か `既存移管` かを確定する
- フォーム運用は `送信先メールアドレス` と `迷惑メール対策` まで決める

## **8. 変更依頼ルール**

変更依頼の扱いが曖昧だと、Phase 4以降の採算が崩れる。

このルールはPhase 3で説明し、承認を取っておく。

### **受付ルール**

- 受付窓口は `メール` または `専用フォーム` のみ
- LINE、DM、口頭での変更依頼は正式受付にしない
- 受領確認は `1営業日以内`
- 有償か無償かの一次判定は `2営業日以内`

### **変更区分**

| 区分 | 例 | 料金ルール | 納期影響 |
| --- | --- | --- | --- |
| 確認・補足 | 営業時間、住所、メニュー名の確定 | 無償 | 原則なし |
| 軽微修正 | 文章差替え、写真差替え、表記修正 | 無償範囲内で対応 | 軽微 |
| 構成変更 | ページ追加、ナビ変更、セクション追加 | 原則有償、`15,000円〜` | あり |
| 機能追加 | 予約、CMS、EC、多言語、会員機能 | 別見積 | 大きい |
| 方向性変更 | 初稿後の全面デザイン変更 | 原則有償 | 大きい |

### **運用ルール**

- 初稿後のデザイン確認は `2回まで` を無償目安にする
- テキスト修正は `3回まで` を無償目安にする
- 仕様凍結後の追加依頼は、着手前に `見積承認` を必須にする
- 値引き対応時は、変更無制限を絶対に付けない
- 追加依頼で納期が動く場合は、費用と日程を同時に再提示する

### **判断の基本線**

- 元の目的を変えない修正は軽微
- ページ数、機能、導線、デザイン方向性が変わるなら別件扱い
- 「ついでに」は採用しない

## **9. リスクレジスター**

| リスク | 早期シグナル | 影響 | 予防策 | 発生時対応 |
| --- | --- | --- | --- | --- |
| 着手金未払い | 支払い日の明言がない | キャッシュフロー悪化、着手遅延 | 入金前着手禁止 | 7日超で停止、30日超で解除判断 |
| 決定権者不在 | 「家族に確認します」が続く | 承認のやり直し | 初回で決定権者確認 | 契約前に承認者同席の場を作る |
| 要望が抽象的 | 「いい感じに」「おしゃれに」中心 | 手戻り増加 | 参考URL必須化 | 方向性を3案提示して選定させる |
| 予算と機能が不一致 | ライトで予約/EC/多言語希望 | 赤字化 | 先に優先順位整理 | スコープ削減か上位プラン提案 |
| 素材提供遅延 | 写真やロゴの提出日が曖昧 | 納期遅延 | 素材締切を明記 | 仮素材運用か納期延期を選ばせる |
| 連絡手段が分散 | LINE、Instagram、メールが混在 | 証跡不足、見落とし | 受付窓口を固定 | 正式依頼は指定窓口へ再送依頼 |
| 期限だけ急ぎ | 1〜2週間で公開希望 | 品質低下、炎上 | 最初に現実ライン提示 | 機能削減か特急費用提示 |
| 権利関係不明の素材 | 拾い画像や他社文章を流用 | 法務リスク | 素材出所確認 | 使用停止、差替えを要求 |
| 解約時の揉め事 | 途中で温度感が急落 | 返金・権利トラブル | 精算条件を契約明記 | 作業ログを基に精算する |

## **10. Phase 4への引き渡し**

Phase 4へ渡す情報は、開発者が迷わない順番で整理する。

情報が多いことより、**不足がないこと** を優先する。

### **引き渡しパックの中身**

- プロジェクトID
- 顧客名、担当者名、主要連絡先
- 契約ステータス、入金ステータス
- 確定プランと金額
- 公開希望日
- 確定ページ一覧
- 機能一覧 `必須 / 任意 / 対象外`
- デザイン方向性、参考URL、NG表現
- 素材一覧 `受領済み / 未受領 / 仮置き`
- ドメイン、フォーム、CMS、保守の決定内容
- 要確認事項 `高 / 中 / 低`
- 変更依頼ルールの共有済みステータス

### **Phase 4へ渡してよい条件**

- 電子署名完了
- 着手金入金確認済み
- 高優先度の曖昧項目がゼロ
- ページ数と機能数が確定
- 参考デザインの方向性が確定
- 素材不足時の代替方針が確定
- 納期に影響する未解決事項がない

### **渡してはいけない状態**

- 「細かいことは作りながら決める」
- 「写真はたぶん後で届く」
- 「予約機能を付けるか未定」
- 「誰が最終承認するか不明」
- 「予算はあとで相談」

## **11. 30 / 60 / 90日導入ロードマップ**

Phase 3運用は、最初から完璧に自動化しなくてよい。

一人運営では、まずテンプレートを固定し、次に判断を標準化し、最後に短縮する。

### **0〜30日**

- 契約書テンプレートを1本に固定する
- 見積書テンプレートを1本に固定する
- ヒアリングフォームを作る
- 変更依頼の受付窓口を1つに絞る
- 「入金前に着手しない」を徹底する

### **31〜60日**

- ヒアリング回答をAIで要件整理する運用を定着させる
- 要件定義サマリーのフォーマットを固定する
- 着手条件、停止条件、解除条件を定型文にする
- 連絡遅延、素材遅延、予算超過のパターンを記録する

### **61〜90日**

- 保守費の定期決済を導入する
- 変更依頼の有償化ルールを実績ベースで調整する
- Phase 3の所要日数を案件ごとに計測する
- Phase 4開始後の手戻り原因を分析し、ヒアリング項目に反映する

### **90日時点で目指す状態**

- 契約から着手までの流れを `半自動` で回せる
- 例外案件を見抜く判断が早くなる
- Phase 4での仕様ブレが大幅に減る
- 保守契約への接続率が安定する

## **14. 運用原則**

Phase 3で守るべき原則はシンプルである。

- 契約前に約束しすぎない
- 入金前に作り始めない
- 曖昧なまま凍結しない
- 変更を善意で飲み込まない
- Phase 4の迷いをPhase 3で潰す

このフェーズを厳密に運用できるほど、一人運営でも案件数を増やして崩れにくくなる。
