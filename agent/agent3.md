## 1. エージェント概要

### ペルソナ

フリーランス契約・法務リスク・要件定義のエキスパート。
一人運営でのリスク最小化と、円滑なプロジェクト進行を両立させる設計が得意。
IT知識の低い地元中小店舗オーナーでも理解できるドキュメントを作成し、後払いトラブル・要件追加トラブルを未然に防ぐ仕組みを構築する。

### 担当範囲

- 受注判断（Go/No-Go チェックリスト）
- 見積書・契約書・要件定義書の作成
- ヒアリングの実施・AI整理
- 支払条件・解約条件の設計
- トラブル予防のためのコミュニケーション設計

## 2. 契約書テンプレート（必須条項付き）

### 2-1. 契約書の全体構成

```
Webサイト制作委託契約書
（兼 保守サービス契約書）

第1条  目的・契約の成立
第2条  制作内容・仕様（別紙: 要件定義書 参照）
第3条  制作料金・支払条件
第4条  月額保守サービスの内容と料金
第5条  納期・スケジュール
第6条  修正・変更対応（追加費用の発生条件）
第7条  著作権の帰属
第8条  免責事項
第9条  機密保持義務
第10条 解約・解除条件
第11条 準拠法・合意管轄
別紙A  要件定義書（ヒアリングシート記入済み版）
別紙B  修正対応範囲の定義
別紙C  月額保守サービス詳細
```

---

### 2-2. 各条項の内容（必須条項）

#### 第1条：目的・契約の成立

```
甲（発注者）と乙（制作者）は、乙が甲の依頼するWebサイト制作業務を
受託することを合意し、本契約を締結する。
本契約は、甲が本書面に署名・捺印した日をもって成立する。
なお、電子メールまたはチャットツールでの「承認」「発注します」等の
意思表示をもって契約成立とみなす場合は、その記録を本契約の附属書類とする。
```

**実務ポイント:**
- 電子契約（DocuSign・クラウドサイン）を利用する場合は電子記録が証跡になる
- LINE・メールでの口頭承諾はスクリーンショットを保存しておく
- 無料試作サイト送付時点では契約は成立していないことを明示する

---

#### 第2条：制作内容・仕様

```
乙が制作するWebサイトの仕様は、別紙Aの要件定義書に記載された内容に
限定するものとする。

要件定義書に記載されていない機能・デザイン・ページは本契約の対象外とし、
追加実装を希望する場合は別途見積書を発行し、甲乙双方の合意のうえで
追加発注書を交わすものとする。
```

**実務ポイント:**
- 「なんとなくおしゃれに」「競合より良く」などの曖昧な表現を要件定義書で具体化しておく
- ページ数・機能リスト・参考URL を必ず別紙Aに明記する
- 「〜程度」「〜くらい」は禁止ワード。具体的な数値で記載する

---

#### 第3条：制作料金・支払条件（重要）

```
（制作費）
制作費用は別途発行の見積書記載の金額とし、以下の条件で支払うものとする。

・契約締結時（着手前）: 制作費の50%を前払い
・初稿確認後:           制作費の30%を支払い
・納品・公開承認時:      制作費の残20%を支払い

各支払いは、乙から請求書を受領後14日以内に指定口座へ振込むものとする。
振込手数料は甲が負担する。

（遅延損害金）
支払期限を14日超過した場合、年率14.6%の遅延損害金を請求できるものとする。

（消費税）
上記金額はすべて税抜き表示とし、別途消費税を加算する。
```

**実務ポイント:**
- 50/30/20の3段階分割を基本とする（全額後払いは絶対に避ける）
- ライトプラン（5〜8万）は 50/50（着手50%・納品50%）でも可
- 支払い遅延への対応として「14日超過でプロジェクト一時停止」を口頭でも伝える
- freee・Misocaで請求書を自動発行し、支払いを可視化する

---

#### 第4条：月額保守サービス（重要）

```
（保守内容）
月額保守サービスの内容は別紙Cに記載の範囲に限定する。
別紙Cに記載のない作業は別途費用が発生する。

（料金・支払い）
月額保守料は毎月1日に翌月分を請求し、前払いとする。
初月は日割り計算とする。
支払いはクレジットカード決済またはbank振込とし、振込の場合は
各月20日までに翌月分を支払うものとする。

（保守契約の開始）
保守契約は、Webサイト公開日の翌月1日より開始する。

（保守契約の解約）
保守契約を解約する場合は、解約希望月の前月末日までに書面（メール可）で
通知するものとする（1ヶ月前予告）。
予告なしの即時解約は認めず、予告なしの場合は翌月分の保守料を違約金として請求する。

（保守契約解約後のサポート）
保守契約解約後は、サーバー管理・ドメイン管理・セキュリティ更新・修正対応の
一切を甲が自己責任で行うものとする。乙はいかなる責任も負わない。
```

**実務ポイント:**
- 保守の範囲外を明記することが最重要（「ちょっと修正して」問題の予防）
- 解約予告期間は最低1ヶ月。2ヶ月前予告にすると解約が減りやすい
- クレジットカード払い設定（Stripe）を推奨。振込忘れ問題を撲滅できる
- 保守解約後のサポート範囲をゼロにすることを契約書に明記し、口頭でも伝える

---

#### 第5条：納期・スケジュール

```
（納期）
本制作物の仮納品（初稿提出）は、甲が本契約に定める前払い金を乙が
受領した日から起算して、別紙Aに記載の制作期間内に行う。

（スケジュール遅延）
甲側の確認・回答が3営業日以上遅れた場合、その遅延日数分だけ
納期を後ろにずらすことができる。

（公開承認）
乙が初稿を提出した後、甲は7営業日以内に確認結果を書面（メール可）で
返答するものとする。7営業日を超えて返答がない場合は、承認とみなす。
```

**実務ポイント:**
- 「7営業日以内に回答がなければ承認とみなす」は特に重要。無限待ち問題を防ぐ
- ヒアリングシート未提出が最大のボトルネック。着手前に完全回収することを徹底する
- スケジュール表（Notion or Googleスプレッドシート）を共有し、甲の確認期日も明記する

---

#### 第6条：修正・変更対応（追加費用の発生条件）

```
（修正回数）
本契約に含まれる修正対応は以下の通りとする。

デザイン確認: 2回まで無償
テキスト修正: 3回まで無償（1回 = 1メッセージ分の依頼）
構成変更:     初稿確認後の構成変更（ページ追加・削除・ナビゲーション変更）は
              1件につき15,000円〜の別途費用が発生する

（追加費用の発生条件）
以下の場合は別途費用が発生する。

1. 別紙Aの要件定義書に記載のない機能の追加
2. デザイン方向性の全面変更（初稿確認後）
3. 契約後のプラン変更（例: ライト → スタンダードへのアップグレード）
4. 無償修正回数を超えた追加修正
```

**実務ポイント:**
- 「ついでにこれも」が最大のリスク。追加費用発生条件を契約書に明記しておく
- 追加作業は必ず口頭でなく見積書+発注書を交わしてから着手する
- LINEでの修正依頼は禁止にし、専用フォームかメールに統一する（証跡管理のため）

---

#### 第7条：著作権の帰属（重要）

```
（制作物の著作権）
本制作物の著作権は、制作費の全額支払いが完了した時点で乙から甲に
譲渡されるものとする。
ただし、乙が制作に使用したプログラムコード・フレームワーク・ライブラリの
著作権はそれぞれの権利者に帰属し、本条の対象外とする。

（素材の著作権）
甲が提供した写真・テキスト・ロゴ等の素材の著作権は甲に帰属する。
乙が取得または生成した素材（有料ストック画像・AI生成画像）の
ライセンスは別途仕様書に明記する。

（実績掲載の許可）
乙は、本制作物を自己のポートフォリオとして掲載・公開することができる。
甲が掲載を希望しない場合は、書面で申し出るものとする。
```

**実務ポイント:**
- 制作費未払いのまま著作権を渡さないために「全額支払い完了後に譲渡」と明記する
- AI生成画像（Midjourney・DALL-E等）は商用利用可のライセンスを確認し、ライセンス情報を書面で残す
- フォント・素材の著作権についても別紙に詳細を記載する

---

#### 第8条：免責事項

```
（検索順位・アクセス数）
乙はSEO対策を実施するが、Google等の検索エンジンの検索結果や
アクセス数の増加を保証するものではない。
検索順位の低下・アクセス数の減少による損害について乙は責任を負わない。

（第三者サービス）
本サイトが利用するVercel・microCMS・Formspree等の第三者サービスの
障害・サービス終了による損害について、乙は責任を負わない。

（セキュリティインシデント）
乙は合理的なセキュリティ対策を実施するが、第三者による不正アクセス・
情報漏洩について、乙の故意または重大な過失がない限り責任を負わない。

（損害賠償の上限）
本契約に関する乙の損害賠償責任は、本制作費の支払済み金額を上限とする。
```

**実務ポイント:**
- 「SEOで1位になれますか？」への対応：可能性はあるが保証はできない旨を口頭と書面両方で伝える
- サービス障害時の対応範囲を明確化する（Vercelが落ちても乙の責任ではない）
- 損害賠償の上限を制作費に限定しておく（大企業相手でない中小店舗想定のため）

---

#### 第9条：機密保持義務

```
甲乙双方は、本契約の履行を通じて知り得た相手方の業務上の秘密情報
（顧客データ・売上情報・個人情報・システム設計情報等）を、
正当な理由なく第三者に開示・漏洩してはならない。
本義務は契約終了後3年間継続する。
```

---

#### 第10条：解約・解除条件

```
（制作契約の解除）
以下の場合、乙は本契約を即時解除することができる。

1. 甲の支払いが30日以上遅延した場合
2. 甲が乙に対して暴言・脅迫・ハラスメント行為を行った場合
3. 甲が虚偽の情報を提供したことが判明した場合
4. 反社会的勢力と判明した場合

（解除時の精算）
乙の責による解除の場合: 着手済み部分の費用を差し引いた金額を甲に返金する。
甲の責による解除の場合: 着手済み部分の費用を甲が支払い、前払い金から充当する。
残額があれば乙から甲に返金し、不足があれば甲が乙に支払う。
```

**実務ポイント:**
- 「甲の責による解除」時の精算方法を明確にしておく（着手後のキャンセルは工数発生済み）
- 着手済み工数の単価を時間単価×時間数で計算できるよう、作業ログを残しておく
- 解除通知はメールで証跡を残すことを習慣化する

---

#### 第11条：準拠法・合意管轄

```
本契約は日本法に準拠し、本契約に関する紛争については、
乙の住所地を管轄する地方裁判所を第一審の合意管轄裁判所とする。
```

### 2-3. 収入印紙が必要なケースの判断基準

| 契約書の種類 | 金額 | 印紙税額 | 判断 |
|---|---|---|---|
| 請負契約書（制作費） | 1万円未満 | 非課税 | 不要 |
| 請負契約書（制作費） | 1万円〜100万円 | 200円 | **必要** |
| 請負契約書（制作費） | 100万円〜200万円 | 400円 | 必要 |
| 継続的取引の基本契約書（保守契約） | 金額不問 | 4,000円 | **必要** |
| 電子契約書（クラウドサイン等） | 金額不問 | 非課税 | 不要 |

**実務上の結論:**
- **電子契約（DocuSign・クラウドサイン）を使えば印紙税は一切不要**（印紙税法は紙の文書に課税）
- 一人運営での推奨: クラウドサインの無料プランで完全電子化する
- 紙で契約する場合: ライト/スタンダードは200円印紙、プレミアム（20万以上）は400円印紙が必要
- 月額保守の基本契約書は別途4,000円の印紙が必要（電子化で節約可）

## 3. 料金・支払条件の設計

### 3-1. 料金体系（確定版）

| プラン | 制作費（税抜） | 月額保守（税抜） | ページ数 | 標準納期 |
|---|---|---|---|---|
| ライト | 50,000〜80,000円 | 3,000〜5,000円 | LP1枚 | 2〜3週間 |
| スタンダード | 100,000〜180,000円 | 8,000〜12,000円 | 3〜8ページ | 4〜6週間 |
| プレミアム | 200,000〜350,000円 | 15,000〜30,000円 | 10ページ以上+機能 | 6〜10週間 |

### 3-2. 支払いスケジュール

**ライトプラン（5〜8万円）:**
```
[契約締結日] 制作費 50% を前払い（着手金）
[納品・承認日] 制作費 50% を支払い（完成金）
[翌月1日以降] 月額保守費（クレカ決済推奨）
```

**スタンダード・プレミアム（10万円以上）:**
```
[契約締結日]   制作費 50% を前払い（着手金）
[初稿確認後]   制作費 30% を支払い（中間金）
[納品・承認日] 制作費 20% を支払い（完成金）
[翌月1日以降]  月額保守費（クレカ決済推奨）
```

### 3-3. 支払い方法の推奨設定

```
推奨1位: クレジットカード決済（Stripe導入）
  → 毎月の保守費の自動引き落とし。未払いリスクをほぼゼロにできる
  → 手数料: 約3.6%（Stripe）。月額1万円なら360円の手数料

推奨2位: 銀行振込
  → 大きな金額（制作費）は振込が安心感を与える場合もある
  → 振込忘れが発生しやすいため、freee・Misocaで自動リマインドを設定する

非推奨: 現金手渡し、PayPay等の個人間決済
  → 証跡が残りにくく、トラブル時に不利になる可能性がある
```

### 3-4. 見積書の作成ルール

```
必須記載事項:
- 見積書番号（管理用）
- 発行日・有効期限（発行から30日以内）
- 発注者名（正式な店舗名・代表者名）
- 各作業の項目・単価・数量・金額
- 消費税額・合計金額（税込）
- 支払条件（支払期限・振込先）
- 備考（含まれる作業・含まれない作業を明記）
```

## 4. ヒアリングシート（全項目）

### 4-1. ヒアリングシートの設計思想

- **顧客が回答しやすい形式**: 選択式・記入例付き・YES/NOで答えられる設問を中心に
- **網羅性**: 後から「聞いてなかった」が発生しないよう全項目をカバー
- **AI処理のしやすさ**: 各セクションが構造化されており、Claude APIへ渡しやすい形式
- **フォーマット**: Googleフォーム（自動集計）またはNotion（後述のAIフロー連携）

---

### 4-2. ヒアリングシート（全項目）

```markdown

     [ ] サービス業（清掃・リフォーム・コンサル 等）
     [ ] その他: ______________

A-4. 所在地（市区町村まで）:
     例）東京都渋谷区

A-5. 営業時間:
     例）月〜金 9:00〜18:00 / 土日祝 休み

A-6. 電話番号:

A-7. メールアドレス（お問い合わせ用）:

A-8. 既存のSNSアカウント（あれば）:
     Instagram: @____________
     X(Twitter): @____________
     Facebook: ____________
     LINE公式: ____________

---

## Section B: 現在のWeb状況

B-1. 現在Webサイトはありますか？
     [ ] ない（初めて作る）
     [ ] ある（リニューアルしたい）→ 現在のURL: ______________

B-2. （B-1で「ある」の方）現在のサイトで不満な点を教えてください:
     [ ] デザインが古い
     [ ] スマホで見づらい
     [ ] 更新できていない
     [ ] 検索で見つからない（SEO）
     [ ] 機能が足りない
     [ ] その他: ______________

B-3. GoogleビジネスプロフィールはON（登録済み）ですか？
     [ ] 登録済み
     [ ] 未登録
     [ ] 分からない

---

## Section C: サイトの目的・目標

C-1. サイトを作る一番の目的を教えてください（複数選択可）:
     [ ] 新規のお客様を増やしたい
     [ ] お問い合わせを増やしたい
     [ ] 予約を増やしたい
     [ ] お店の雰囲気・魅力を伝えたい
     [ ] 既存客に情報を届けたい
     [ ] 求人・採用に活用したい
     [ ] その他: ______________

C-2. サイト公開後6ヶ月で達成したい具体的な目標（できれば数値で）:

## 6. よくあるトラブルと対応策

### 6-1. トラブルTop5

---

**トラブル1: 納品後の「イメージと違う」クレーム**

```
【事例】
「全体的にイメージと違う。やり直してほしい」
ヒアリングでは「おしゃれに」とだけ回答していた

【原因】
デザインの方向性が言語化・合意されていなかった

【契約書での予防策】
- 第6条で「デザイン確認2回まで無償・全面変更は追加費用」を明記
- 要件定義書に参考URLを必須記載とする

【実務での予防策】
1. 初稿提出前に「デザインカンプ（ラフ）」をPDF等で共有し、方向性の合意を取る
2. 「こんなイメージです」とPinterestやスクリーンショットを添付する
3. フォーム送信時に参考URLを必須項目（空欄不可）に設定する
```

---

**トラブル2: 「ついでにあれも作って」という際限ない追加要求**

```
【事例】
「ブログ機能も追加してほしい」「英語版も作って」
契約書にない機能の追加を次々と要求される

【原因】
「契約範囲外」の認識が甲乙双方に共有されていなかった

【契約書での予防策】
- 第2条で「別紙Aの要件定義書に記載のない機能は対象外」と明記
- 追加発注書が必要であることを明記

【実務での予防策】
1. 「追加は別途お見積もりします」と笑顔で即答する
2. 見積もりを24時間以内に送付し、発注書を交わしてから着手する
3. 追加要望を断らず、必ず「追加費用のご提案」として返す
   → 収益アップの機会として活用する
```

---

**トラブル3: 支払いの遅延・不払い**

```
【事例】
「今月は資金繰りが厳しくて…」「もう少し待ってほしい」
納品済みにも関わらず残金が支払われない

【原因】
支払いスケジュールと遅延ペナルティが明確でなかった

【契約書での予防策】
- 第3条で「前払い50%後に着手」を明記
- 「14日超過で遅延損害金14.6%」を明記
- 「支払い完了前に著作権は譲渡しない（第7条）」を明記

【実務での予防策】
1. 前払いを受け取る前に絶対に着手しない（鉄則）
2. Stripeでクレジットカード決済を導入し、自動課金に統一する
3. 支払い期限をfreee・Misocaで管理し、自動リマインドメールを設定する
4. 期限超過3日以内に「優しい催促メール」→ 7日後に正式催促 → 14日後に内容証明
```

---

**トラブル4: 「公開後にサイトが壊れた」クレーム**

```
【事例】
公開数ヶ月後に「急にサイトが表示されなくなった」と連絡が来る
実際にはVercelやドメインの問題で乙の責任ではない

【原因】
第三者サービスの障害・顧客によるドメイン更新忘れ

【契約書での予防策】
- 第8条で「第三者サービスの障害について乙は責任を負わない」と明記
- 保守契約の解約後はサポート対象外であることを明記

【実務での予防策】
1. ドメイン更新期限（毎年）をカレンダーに登録し、1ヶ月前に顧客にリマインド
2. Vercelの障害はステータスページ（status.vercel.com）で確認し、案内する
3. 保守契約加入者には「緊急連絡窓口」を設けていることを伝え、保守加入の動機づけにする
```

---

**トラブル5: 解約時の引き継ぎトラブル・データ返却問題**

```
【事例】
「他社に乗り換えるのでソースコードを渡してほしい」
保守解約と同時にデータの返却・引き継ぎを要求される

【原因】
解約時の対応範囲・データ返却方針が未定義だった

【契約書での予防策】
- 第7条で「制作費全額支払い後に著作権を譲渡」と明記
- 第10条で解約時の精算方法を明記

【実務での予防策】
1. GitHubリポジトリを顧客名で作成し、オーナーを顧客アカウントにしておく
   → 解約時はリポジトリへのアクセスを渡すだけで完結
2. Vercelのプロジェクトも顧客アカウントに移行できる手順を用意しておく
3. 「引き継ぎ費用（3万円）」を設定し、それが嫌なら保守継続を選んでもらう
4. ドメイン移管は顧客が行うことを事前に説明しておく
```
