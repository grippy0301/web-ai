
# Web制作サービス Phase 5 ロードマップ（デプロイ・納品・QA）_JP

> 作成日: 2026-05-21  
> 対象: ひとり運営のWeb制作事業者  
> 参照元: `new/web_creation_service_roadmap_JP.md` の Phase 5、`new/agents/05_deploy_ops.md`  
> 位置づけ: Phase 4で完成したサイトを、安全に公開し、品質保証を通し、納品可能な状態にして Phase 6 の保守運用へ接続するための実行ロードマップ

---

## 1. フェーズ概要

Phase 5 の目的は、単にサイトを公開することではない。ひとり運営でも事故なく再現できる形で、`開発完了 → 公開判定 → 本番反映 → 顧客引き渡し → 保守移管` を一本の運用フローとして固めることにある。

このフェーズでは、次の3つを同時に満たす必要がある。

1. 本番公開しても壊れないこと
2. 顧客が受け取って運用を始められること
3. 自分が Phase 6 で無理なく保守できること

推奨タイムボックスは以下。

| 項目 | 目安 |
|---|---|
| 技術設定・デプロイ | 0.5〜1時間 |
| QA実施 | 2〜3時間 |
| 納品資料整備 | 0.5〜1時間 |
| 引き渡しデモ | 60〜90分 |
| DNS/SSL待ち時間 | 15分〜48時間 |

このフェーズの出口条件は、`本番URLで正常公開済み`、`品質基準を通過済み`、`顧客向け納品物一式が揃っている`、`Phase 6 に必要な運用情報が整理済み` の4点である。

---

## 2. 成功指標

Phase 5 は「なんとなく公開できた」ではなく、次の指標で合否を判断する。

| 指標 | 目標 | 出荷最低ライン | 判断ルール |
|---|---|---|---|
| `npm run build` | 常時成功 | 必須 | 失敗時は Phase 4 に差し戻し |
| Vercel本番デプロイ | 1回で `Ready` | 必須 | 連続失敗2回で原因切り分けが終わるまで公開停止 |
| Lighthouse Performance | 90+ | 80+ | 80未満は公開停止 |
| Lighthouse Accessibility | 95+ | 90+ | 90未満は公開停止 |
| Lighthouse Best Practices | 95+ | 90+ | 90未満は公開停止 |
| Lighthouse SEO | 100 | 95+ | 95未満は公開停止 |
| セキュリティヘッダー | A相当 | B相当以上 | 重要ヘッダー欠落は公開停止 |
| `npm audit` | Critical/High 0件 | Critical/High 0件 | 例外運用しない |
| 初回手戻り件数 | 0〜1件 | 2件以下 | 3件以上なら Phase 4 テンプレート改善対象 |
| 納品デモ時間 | 60〜90分 | 120分以内 | 120分超なら資料か手順が不足 |

追加の運用KPIとして、公開後7日以内の重大障害ゼロ、顧客が7日以内に自力で1件更新できることを Phase 6 への接続指標とする。

## 3. 前提条件

Phase 5 に入る前に、以下が揃っていない場合は開始しない。

| 前提 | 必須状態 |
|---|---|
| コード | Phase 4 の実装完了、主要ページ完成、未解決の重要バグなし |
| リポジトリ | Git 管理済み、`main` `staging` `develop` の役割が明確 |
| ビルド定義 | `package.json` の build コマンド確定、`dist/` 出力確認済み |
| 環境変数 | `.env.example` があり、本番・プレビューで必要値が明確 |
| CMS | microCMS のスキーマ確定、初期コンテンツ入力済み |
| ドメイン | 本番ドメイン方針確定、Cloudflare 管理可否が明確 |
| 承認 | 顧客が staging 確認を終え、公開Goが出ている |
| 素材・文言 | 画像・ロゴ・営業時間・住所・連絡先が確定 |
| トラッキング | GA4、Search Console の要否が確定 |
| 納品責任 | 誰が何のアカウントを保有し続けるか決定済み |

意思決定ルール:

- 顧客承認がない状態で `main` へ反映しない。
- ドメイン運用主体が曖昧なら公開を遅らせても先に権限整理を行う。
- 素材や文言が未確定でも staging までは進めてよいが、本番公開は止める。

## 4. スコープと非対象

### スコープ

- GitHub と Vercel の連携設定
- 本番 / staging / preview のデプロイ導線整備
- Cloudflare DNS と SSL の設定
- Lighthouse、SEO、セキュリティ、レスポンシブのQA
- 納品資料の作成
- 顧客への引き渡しデモ
- Phase 6 に必要な運用情報の整備

### 非対象

- 大幅なデザイン改修
- 追加機能の新規開発
- 本格的なSEO施策の実行
- 広告運用、SNS運用、LP改善のPDCA
- 顧客社内のIT教育全般
- メールサーバー移管や業務システム統合

判断ルール:

- Phase 5 中に新規機能要望が出た場合は、原則として `公開後対応` に回す。
- 品質に直結しない見た目の細部修正は、公開延期の理由にしない。
- セキュリティ、リンク切れ、フォーム不達、法的表示欠落は見た目より優先する。

## 5. デプロイワークフロー

Phase 5 は、以下の6段階で進める。ひとり運営では「いつでも出せる状態」を目指すより、「止める基準が明確な順送り」の方が強い。

### Step 1. リリース凍結と受け取り確認

やること:

- Agent 4 から受け取ったリポジトリをローカルで `npm ci` `npm run build` まで確認
- `package.json`、`.env.example`、CMS接続情報、公開ドメインを確認
- 未確定事項を `公開前確認リスト` として1枚に集約

意思決定ルール:

- ローカルビルド失敗時はこの時点で停止
- `.env.example` がない案件は、その場で整備しない限り先に進まない
- 外部サービス依存が3件以上ある場合は、接続先ごとに疎通責任者を明記する

### Step 2. Vercel プロジェクトとブランチ導線の固定

やること:

- GitHub リポジトリを Vercel に接続
- `develop` を内部確認用、`staging` を顧客確認用、`main` を本番用に固定
- `main` への直接 push を避け、`staging` 承認後にのみ本番反映する

推奨運用:

| ブランチ | 用途 | 誰のための環境か |
|---|---|---|
| `develop` | 実装確認 | 自分 |
| `staging` | 顧客確認 | 顧客 + 自分 |
| `main` | 本番公開 | エンドユーザー |

意思決定ルール:

- `staging` 未確認で `main` に出さない
- 本番直前に仕様変更が入った場合、先に `staging` を更新して再承認を取る
- 急ぎ案件でも `main` 直接修正はしない。事故時に戻れなくなるため

### Step 3. 環境変数と本番設定の確定

やること:

- Vercel の Production / Preview に環境変数を分離登録
- `.env.local` はローカル専用とし、Git へ含めない
- `vercel.json` の SPA ルーティング、ヘッダー、キャッシュ方針を確認

意思決定ルール:

- `VITE_` 付き変数であっても秘匿すべき値はフロント公開前提を再確認する
- APIキーがブラウザ配信に不向きなら設計見直し。Phase 5 で無理に通さない
- CSP は「最低限守る」方針で開始し、表示崩れを起こすほど厳しくしすぎない

### Step 4. DNS・ドメイン・SSL の切り替え

やること:

- Vercel にカスタムドメインを追加
- Cloudflare で `www` は CNAME、ルートは A レコードを設定
- SSL 発行と `Valid Configuration` を確認
- HTTP→HTTPS、必要ならルート→`www` リダイレクトを確認

意思決定ルール:

- Cloudflare プロキシは証明書安定まで `DNSのみ` を維持する
- DNS 伝播が60分を超えても安定しない場合、当日公開をいったん見送り再確認する
- 公開期限が厳しい場合でも、顧客合意なしに Vercel サブドメインを本番扱いしない

### Step 5. QA 実施と公開判定

やること:

- `staging` と本番候補URLで QA を実施
- Lighthouse、SEO、セキュリティ、フォーム、主要導線、端末表示を確認
- 結果を `公開可 / 条件付き公開 / 公開不可` の3段階で判定

意思決定ルール:

- 条件付き公開が許されるのは、軽微な改善のみ
- 問い合わせ導線、店舗情報、営業時間、地図、電話リンクの不備は公開不可
- 代替手段がない重大バグは、納期より品質を優先して公開延期

### Step 6. 本番反映、納品、初動監視

やること:

- `main` 反映後に本番URLで再チェック
- 納品資料一式をまとめて送付
- 60〜90分の引き渡しデモを実施
- 翌日と1週間後にフォローアップ

意思決定ルール:

- 顧客が初期パスワードを変更していない状態では「完全納品」にしない
- デモで顧客が更新操作を1回も実行できない場合、資料を見直して再説明を入れる
- 公開後24時間以内に最低1回は自分で本番URLの再確認を行う

## 6. QAゲート

QA はまとめて眺めるのではなく、ゲート制で潰す。ひとり運営では「全部見たつもり」が最大の事故要因になる。

### Gate 1. ビルド・デプロイゲート

通過条件:

- `npm run build` 成功
- `dist/` 出力確認
- Vercel デプロイが `Ready`
- SPA ルーティングと 404 導線が成立

止める条件:

- build エラー
- 環境変数未反映
- microCMS データ未表示

### Gate 2. パフォーマンス・アクセシビリティゲート

通過条件:

| 指標 | 基準 |
|---|---|
| Performance | 80以上、理想90以上 |
| Accessibility | 90以上、理想95以上 |
| Best Practices | 90以上、理想95以上 |
| SEO | 95以上、理想100 |
| LCP | 2.5秒未満 |
| CLS | 0.1未満 |

止める条件:

- モバイル Performance 80未満
- Accessibility 90未満
- 画像やフォントの最適化不足で主要ページが重い

改善優先順位:

1. Hero画像最適化
2. 不要JS削減
3. フォント読み込み最適化
4. CLS原因の寸法未指定修正

### Gate 3. SEO・計測ゲート

通過条件:

- 全主要ページに `title` `description` `canonical` 設定
- OGP / Twitter Card 設定
- `robots.txt` `sitemap.xml` 配置
- 構造化データ設定
- Search Console / GA4 の要否に応じた設定完了

止める条件:

- 店名や地域名が title に入っていない
- `noindex` の混入
- サイトマップ未配置

### Gate 4. セキュリティ・ブラウザゲート

通過条件:

- HTTPS 正常
- 主要ヘッダー設定済み
- `npm audit` Critical/High 0件
- `.env` 系未コミット
- Chrome / Safari(iPhone想定) / Firefox の主要表示OK
- 375px / 768px / 1280px で大崩れなし

止める条件:

- HTTPでアクセス可能なまま
- 本番ソースへ不要なデバッグ情報が残る
- スマホのファーストビューが崩れる

### Gate 5. 納品準備ゲート

通過条件:

- 納品資料一式が揃っている
- 顧客へ渡すURLとアカウント一覧が整理済み
- CMS更新マニュアルが顧客のITリテラシーに合っている
- デモの流れを事前リハーサル済み

止める条件:

- 顧客がどのURLで何をするか説明できない
- ログイン情報の所在が曖昧
- 保守対象と保守対象外が文書化されていない

## 7. 納品パッケージ

納品は「URLを送って終わり」にしない。最低でも次の単位で束ねる。

| 区分 | 内容 | 備考 |
|---|---|---|
| 公開物 | 本番URL、主要ページ一覧 | まず最初に確認させる |
| 管理情報 | microCMS URL、必要に応じて Vercel / Cloudflare 情報 | 初期パスワードはその場で変更 |
| 技術設定 | Search Console、GA4、サイトマップ送信状況 | 未設定なら未設定と明記 |
| 運用資料 | microCMS更新マニュアル、サイト構成説明書、保守説明書 | 顧客の操作レベルに合わせる |
| 連絡経路 | 緊急連絡先、通常依頼窓口、対応時間 | 深夜障害の期待値調整を先にする |
| 引き渡し実施 | 60〜90分のデモ、1回の更新ハンズオン | 顧客が実際に手を動かす |

納品時の実務ルール:

- URL一覧は1ページにまとめ、顧客が迷わないようにする
- パスワードはメール本文に直書きしない。別経路で渡す
- 「保守に含むもの」「別料金になるもの」を必ず文書化する
- 更新マニュアルは機能一覧ではなく「よくある作業順」で書く

## 8. 運用・セキュリティガードレール

ひとり運営では、ルールを減らすより「守るルールを固定」する方が強い。最低限、以下は固定運用にする。

### アカウント・権限

- 顧客保有が望ましいアカウントは、可能な限り顧客名義で作成する
- 自分保有のまま始める場合は、移管可否と期限を明文化する
- 共有パスワードは納品当日に変更してもらう

### リリース運用

- 本番公開は原則として平日の日中に行う
- 金曜夜・連休前の本番反映は避ける
- 本番直前に仕様変更を入れない

### シークレット管理

- `.env` は Git に入れない
- シークレットは Vercel / GitHub Secrets に限定する
- 顧客APIキーをローカルメモ帳やチャットに散らさない

### セキュリティ設定

- HTTP→HTTPS は必須
- 最低限のセキュリティヘッダーを統一
- 使っていない外部スクリプトは載せない
- フォーム送信先、地図埋め込み、解析タグは公開前に実URLで確認する

### ロールバック

- 直前の安定デプロイを Vercel で即復元できるようにする
- ロールバック判断は「問い合わせ不能」「主要コンテンツ欠落」「レイアウト崩壊」で即時
- ロールバック後は、その日のうちに原因を記録する

### 監視と記録

- 公開後24時間以内に再確認
- 最低でも 1日後、1週間後、30日後 の3回は状況を見る
- 障害や再修正は、次回テンプレート改善のために必ず記録する

## 9. リスクレジスター

| リスク | 兆候 | 影響 | 予防策 | 発生時対応 |
|---|---|---|---|---|
| DNS反映遅延 | `Invalid Configuration` が続く | 公開延期 | ドメイン情報を前日までに確定 | 顧客へ即連絡し、再反映待ち |
| SSL未発行 | HTTPS警告 | 信頼毀損 | Cloudflare を `DNSのみ` で設定 | 証明書発行完了まで本番案内停止 |
| 環境変数ミス | CMSが空表示 | 公開後不具合 | Preview/Production 分離管理 | 値を再設定し再デプロイ |
| 画像最適化不足 | モバイルが重い | CV低下 | Hero画像を事前圧縮 | 画像差し替え後に再測定 |
| 顧客の更新不能 | デモ中に迷う | 保守負荷増大 | 低ITリテラシー前提で手順書作成 | 補足マニュアルを追加送付 |
| アカウント責任不明 | 誰が更新するか曖昧 | トラブル長期化 | 管理責任者を納品前に決定 | 役割整理して Phase 6 へ引継ぎ |
| 本番直前の追加要望 | リリース直前の仕様変更 | 納期遅延 | 非対象ルールを事前共有 | 別タスク化して公開優先 |
| フォーム不達 | テスト漏れ | 問い合わせ損失 | 本番URLで必ず送信テスト | 一時的に電話・LINE導線を前面化 |
| 依存パッケージ脆弱性 | `npm audit` High/Critical | セキュリティ事故 | 公開前監査を固定化 | 修正できるまで公開停止 |

## 10. Phase 6 への引き継ぎ

Phase 5 の完了は、Phase 6 の開始準備まで含む。引き継ぎは口頭ではなく、最低限次の情報を1セットにする。

| 引き継ぎ項目 | 内容 |
|---|---|
| 顧客基本情報 | 店舗名、担当者名、連絡先、ITリテラシー |
| 公開URL群 | 本番URL、CMS URL、Search Console、GA4 |
| アカウント責任 | Vercel、Cloudflare、microCMS の所有者 |
| 品質スコア | Lighthouse、セキュリティ評価、確認済み端末 |
| 保守条件 | 月額、対応範囲、軽微修正の上限 |
| 残課題 | 公開後に回した項目、改善候補、アップセル余地 |
| 次回接点 | 翌日、1週間後、1か月後のフォロー予定 |

引き継ぎ判断ルール:

- 上記が揃っていないなら「公開済み」でも Phase 5 完了扱いにしない
- 顧客が自走できない前提なら、Phase 6 初月に操作フォローを組み込む
- アップセル候補は思いつきではなく、実データ確認後に出す

## 11. 30/60/90日ロールアウト

本番公開で終わらせず、最初の90日を保守導入期間として設計する。

### 30日以内: 安定化フェーズ

目的:

- 公開事故を潰す
- 顧客が最低限の更新を自走できる状態にする

やること:

- 翌日フォロー
- 1週間後の操作確認
- Search Console の初期エラー確認
- フォーム到達、電話リンク、地図、主要導線の再確認

判断ルール:

- 顧客が1回も更新していない場合、操作障壁があると見なして補足支援を入れる
- 30日以内の軽微修正は、保守契約範囲内に含めるかを事前ルールで判断する

### 60日以内: 計測・改善準備フェーズ

目的:

- 使われ方を把握し、Phase 6 の運用観点を固める

やること:

- GA4 の流入状況確認
- モバイル比率、離脱しやすいページ、問い合わせ導線の確認
- よくある更新作業を整理し、保守負荷を見積もる

判断ルール:

- 流入はあるがCVが弱い場合、まず導線改善を優先する
- 流入自体が少ない場合、SEOやMEOの提案は60日以降に絞って行う

### 90日以内: 継続運用判断フェーズ

目的:

- 保守継続、改善提案、アップセルの優先順位を決める

やること:

- 月次レポートの初回提出
- 保守プランの適合確認
- 追加提案の可否判断

判断ルール:

- 問い合わせ増加が見えるなら、予約導線やコンテンツ拡張を提案
- 更新頻度が低いなら、まずは運用簡素化を優先する
- 顧客反応が薄い場合、無理に追加提案せず保守価値の可視化から始める

## 13. Done Checklist

Phase 5 完了時は、以下をすべて満たすこと。

```
[ ] 本番URLがHTTPSで正常公開されている
[ ] 主要ページ・主要導線・フォームが本番で動作する
[ ] Lighthouse 基準を満たしている
[ ] SEO必須設定が入っている
[ ] Critical/High の脆弱性がゼロ
[ ] Chrome / Safari / Firefox の主要確認が終わっている
[ ] 納品資料一式が顧客へ渡せる状態
[ ] 顧客がログインし、最低1回の更新操作を体験した
[ ] 初期パスワード変更が完了している
[ ] 保守対象 / 非対象が共有済み
[ ] 翌日・1週間後・30日後のフォロー予定が決まっている
[ ] Phase 6 に渡す運用情報が整理済み
```

## 14. このフェーズの最終判断

Phase 5 の本質は、`公開すること` ではなく `安心して運用開始できる形に整えること` にある。ひとり運営では、公開速度よりも再現性と事故回避の方が長期利益に直結する。

そのため最終判断はシンプルでよい。

- 品質基準を満たすなら公開する
- 顧客が受け取れないなら納品しない
- Phase 6 に引き継げないなら完了扱いにしない

この3つを守れば、Phase 5 は単発の納品作業ではなく、継続収益につながる運用開始フェーズになる。
