
# Web制作サービス Phase 6 ロードマップ: 保守・継続収益・成長

> 作成日: 2026-05-21  
> 対象読者: 一人で保守運営を回すWeb制作者  
> 参照元: `new/web_creation_service_roadmap_JP.md` の Phase 6 / `new/agents/06_growth_manager.md`  
> 目的: 納品後のサイト運営を「都度対応」から「継続収益を積み上げる運営システム」へ変える

---

## フェーズ概要

Phase 6 は、納品済みサイトを安全に維持しながら、保守契約を安定したMRRへ育てるフェーズである。ソロオペレーターにとって重要なのは、保守作業そのものを増やすことではなく、判断基準と作業窓口を標準化し、案件数が増えても破綻しない運営形にすることだ。

このフェーズで作るべき状態は次の3つに集約される。

1. 障害・更新・軽微修正が、週次と月次の定例運用で処理できる。
2. 顧客ごとの価値を毎月見える化し、解約理由を先回りで潰せる。
3. アップセル提案が思いつきではなく、数値トリガーに基づいて行われる。

運用原則は以下の通り。

- 標準化できない作業は自動化しない。
- 保守と追加開発を混同しない。
- 顧客ごとの特例より、全顧客で再利用できる仕組み改善を優先する。
- 毎月のレポートは「成果報告」ではなく「継続価値の証明」として扱う。

## 成功指標

ソロ運用の基準値として、立ち上げ期と安定期の2段階で管理する。

| 指標 | 立ち上げ期の目標 | 安定期の目標 | 判断ルール |
|---|---:|---:|---|
| 保守契約社数 | 5社以上 | 15社以上 | 25社を超えたら新規保守受注を一時調整し、標準化を先に進める |
| 保守MRR | 4万円以上 | 12万円以上 | 固定費をMRRで賄える状態を最優先とする |
| 1社あたり実作業時間（スタンダード相当） | 月60分以内 | 月45分以内 | 2ヶ月連続で超過したら、作業内容か価格設定を見直す |
| 月次レポート期限内送付率 | 90%以上 | 95%以上 | 3営業日以内に送付できない顧客が続いたらレポート構成を削る |
| P1障害の初動時間 | 4営業時間以内 | 2営業時間以内 | 初動は「原因確定」ではなく「受領連絡と切り分け開始」でよい |
| パッチ/マイナー更新の自動処理率 | 60%以上 | 80%以上 | 事故が続くなら自動化率より安全性を優先する |
| 月次解約率 | 5%未満 | 3%未満 | 1四半期で2件以上解約したら、理由を分類して対策を更新する |
| 四半期アップセル成約率 | 10%以上 | 15%以上 | 提案数より「根拠付き提案数」を重視する |

補助指標として、`保守工数 / MRR` も毎月確認する。目安は `1万円あたり45分以内` で、これを超える契約は値付けまたはスコープが崩れている可能性が高い。

## 前提条件

Phase 6 を始める前に、各顧客案件で最低限そろっているべきものを明確にする。ここが欠けていると、保守は仕組み化ではなく「後始末」になる。

### 必須の運用基盤

- GitHub リポジトリが存在し、デプロイ元ブランチが固定されている
- Vercel 等の本番環境が整備され、再デプロイ手順が明文化されている
- Uptime 監視、GA4、Search Console のうち少なくとも監視とアクセス計測が有効
- ドメイン更新日、DNS管理者、請求先、緊急連絡先が台帳に登録済み
- 保守プラン、対応範囲、連絡方法、返信目安が契約か説明書に反映済み
- 軽微修正依頼の受付窓口が1つに集約されている
- 月次レポートの送付先担当者が決まっている
- 請求手段が自動化されているか、少なくとも毎月の請求ルールが固定されている

### 開始判定ルール

- 上記の必須条件が1つでも欠ける案件は「保守開始」ではなく「保守準備中」として扱う
- ドメイン管理者とリポジトリ権限が不明な案件は、保守販売より先に権限整理を行う
- 月次レポートに必要な計測が未設定なら、初月は「運用開始レポート」を送って埋める

## 対象範囲と非対象

保守で何をやるかより、何をやらないかの定義のほうが重要である。ソロ運用では、ここが曖昧だと利益が消える。

### 対象範囲

| 項目 | 標準対応 | 補足 |
|---|---|---|
| ホスティング・ドメイン・SSL維持 | 対象 | 更新期限管理、失効防止、軽微なDNS確認を含む |
| 依存関係更新 | 対象 | パッチ/マイナー中心。メジャーは別判断 |
| 死活監視と障害一次対応 | 対象 | 復旧対応または暫定対応まで |
| 月次レポート送付 | 対象 | 数値要約、所感、次月提案を含む |
| 軽微な文言差し替え・画像更新 | 条件付き対象 | プラン内回数または月間時間枠内で実施 |
| 小さな改善提案 | 対象 | 実装ではなく提案までを基本とする |

### 非対象

| 項目 | 扱い | 判断ルール |
|---|---|---|
| 新規ページ追加 | 別見積もり | ページ追加は保守ではなく小規模開発 |
| 新機能実装・外部連携追加 | 別見積もり | 予約、決済、会員機能、CMS構造変更など |
| 広告運用代行 | 非対象 | 必要なら外部連携や別商品化 |
| SNS運用代行 | 非対象 | 投稿作成・返信代行は含めない |
| 当日中の非緊急修正 | 非対象 | 緊急性の基準は売上毀損・問い合わせ停止の有無で判定 |
| 無制限相談 | 非対象 | 相談窓口は用意しても、実作業はスコープ管理する |

### 依頼の切り分けルール

| 依頼内容 | 扱い | 判断基準 |
|---|---|---|
| 文言修正、画像1〜2点差し替え | 保守内 | 30分以内で終わり、UI構造が変わらない |
| セクション追加、フォーム項目変更、構造変更 | 小規模改修 | 30分超または確認工数が大きい |
| 新ページ、新機能、CMS項目追加 | 別案件 | 要件整理が必要なものは保守外 |

保守内か迷う依頼は、`作業時間` ではなく `再現性` で判断する。同じ依頼が複数顧客で繰り返されるならテンプレート化を検討し、単発で複雑なら小規模改修として切り出す。

## 保守ワークフロー

保守は「依頼が来たら都度やる」方式ではなく、日次・週次・月次の固定サイクルで処理する。ソロオペレーターでは、バッチ処理が最も効く。

### 日次・週次・月次の運用リズム

| 頻度 | 実施内容 | 目安時間 | 目的 |
|---|---|---:|---|
| 毎営業日 朝 | 監視アラート確認、支払い失敗確認 | 10分 | 緊急事項の見落とし防止 |
| 毎営業日 夕方 | 依頼受付確認、翌日の対応枠整理 | 10分 | 割り込みの抑制 |
| 毎週月曜 | Dependabot PR、デプロイログ、未処理依頼の一括確認 | 60〜90分 | 技術保守の定例処理 |
| 毎週木曜 | 軽微修正のまとめ実施 | 60分 | 細かい依頼の分散を防ぐ |
| 毎月1〜3営業日 | レポート生成、所感追加、送付 | 90〜180分 | 価値の可視化とアップセル種まき |
| 四半期末 | KPI見直し、料金・対象範囲・テンプレート更新 | 120分 | 利益率の維持 |

### 障害・依頼の優先度ルール

| 優先度 | 例 | 初動目安 | 対応方針 |
|---|---|---:|---|
| P1 | サイト停止、フォーム送信不能、SSLエラー | 2営業時間以内 | まず暫定復旧。原因分析はその後 |
| P2 | 表示崩れ、軽微な機能不具合、主要文言の誤り | 翌営業日まで | 定例枠外でも早めに処理 |
| P3 | 画像差し替え、文言更新、相談 | 次回定例枠 | 週次・木曜枠にまとめる |

### 受付からクローズまでの流れ

1. 依頼を1つの窓口で受ける  
   電話はP1のみ、通常依頼はメールかフォームに寄せる。
2. `P1 / P2 / P3` と `保守内 / 保守外` をその場で判定する  
   ここを先送りにすると、後で値引き交渉になる。
3. 対応日を即返答する  
   「今週木曜対応予定です」のように枠で答える。
4. 実施後は、内容・時間・次回注意点を台帳に残す  
   後の値上げ判断とアップセル判断に使う。

### 運用上の決定ルール

- 同一顧客から月2回以上の軽微修正が続く場合は、翌月に上位プランを案内する
- 同じ種類の修正を3回繰り返したら、手動処理ではなく運用改善かCMS改善を検討する
- 木曜定例枠で処理しきれない依頼が2週続いたら、保守契約数かスコープが過多と判断する

## 自動化ポリシー

自動化は「手間削減」のためではなく、「判断のばらつきを減らす」ために使う。顧客影響が大きい処理ほど、完全自動ではなくガードレール付き自動化にする。

### 自動化の基本原則

- 15分以内にロールバックできない処理は完全自動化しない
- 顧客向け文面、料金、契約条件は自動送信しない
- 2回連続で失敗した自動化は一度停止し、手動フローへ戻す
- 自動化前に、通知先・ログ・復旧手順の3点をそろえる

### 項目別ポリシー

| 項目 | 標準モード | 自動実行条件 | 手動介入条件 |
|---|---|---|---|
| パッチ/マイナー依存更新 | 半自動 | CI通過、Preview正常、変更が依存関係中心 | 本番表示差分、主要UI崩れ、監視異常 |
| メジャーアップデート | 手動 | なし | 毎月1回のまとめ検証で判断 |
| 本番デプロイ | 半自動 | main マージ後に自動実行 | リリースノートが必要な変更、フォーム周りの変更 |
| 死活監視 | 完全自動 | 5分間隔監視、通知二重化 | 誤検知が月2回を超えたら閾値見直し |
| ドメイン更新通知 | 完全自動 | 60日前、30日前通知 | 管理者不明、請求失敗時 |
| 月次レポート草案生成 | 自動 | データ取得成功、先月比較可能 | データ欠損、異常値、重要顧客 |
| レポート送付 | 半自動 | 最終確認後に一括送付 | 低調顧客、アップセル提案を添える顧客 |
| 請求失敗リマインド | 半自動 | 失敗3日後に自動通知 | 7日以上未入金、返信なし |

### Dependabot 運用ルール

- パッチ/マイナー更新は、`テスト通過` `Lighthouse基準維持` `Previewで致命的不具合なし` を満たした場合のみマージ候補にする
- メジャー更新は定例日以外に触らない
- 月内に未処理PRが10本を超えたら、新規改善ではなくメンテナンス処理を優先する

## 月次レポート運用フロー

月次レポートは、単なるアクセス数の送付ではない。顧客に「保守費を払う理由」を毎月再確認してもらうための運用接点である。

### 送付フロー

| タイミング | 作業 | 出力 | 判断ルール |
|---|---|---|---|
| 1営業日目 午前 | GA4等のデータ取得 | レポート草案 | データ未取得なら当日中に再実行 |
| 1営業日目 午後 | 異常値確認、前月比較 | 要確認顧客リスト | 前月比のみで語れない顧客は3ヶ月平均を使う |
| 2営業日目 | 所感追記、次月施策決定 | 送付版レポート | 提案は1通につき1テーマまで |
| 3営業日目まで | 一括送付、台帳記録 | 送付ログ | 未送付ゼロで締める |

### レポートの標準構成

1. 今月のハイライト  
   セッション、問い合わせ、主要流入、モバイル比率など
2. 数値の解釈  
   何が良かったか、何が停滞しているかを短く説明
3. 次月の保守予定  
   更新、点検、改善予定を1つ示す
4. 必要時のみアップセル提案  
   1文から3文で十分。押し売りしない

### レポート文章の判断ルール

- セッションが100未満の顧客には、前月比の小さな増減を大げさに扱わない
- 問い合わせゼロが続く顧客には、流入改善か導線改善のどちらが先かを決めてから書く
- 数字が弱い月でも、保守価値が伝わる項目を必ず1つ入れる  
  例: 障害ゼロ、更新実施、表示速度維持、セキュリティ更新
- コメントは長文にせず、顧客が返信しやすい一言で締める  
  例: 「必要であれば来月は予約導線の改善案もお出しできます」

### 送付遅延時のルール

- 3営業日以内に本レポートを送れない場合は、4営業日目に簡易版を先に送る
- 何も送らない月を作らない
- データ欠損がある場合でも、運用報告と次の対応だけは送る

## アップセル戦略

アップセルは「売り込み」ではなく、「顧客の伸びしろに応じて次の打ち手を提示すること」と定義する。ソロ運用では、提案機会を増やすより、勝率の高い提案だけに絞るほうがよい。

### 段階別ロードマップ

| 顧客フェーズ | トリガー | 提案内容 | 価格帯の目安 |
|---|---|---|---|
| 公開直後〜3ヶ月 | 軽微修正が月2回超、相談頻度が高い | ライト→スタンダードへの保守アップグレード | 月+5,000〜8,000円 |
| 3〜6ヶ月 | 月300セッション超なのに問い合わせが少ない | SEO強化、ブログ/FAQ追加、導線改善 | 初期3〜5万円 + 月1〜2万円 |
| 6〜12ヶ月 | モバイル比率70%以上、電話依存が高い | 予約導線、フォーム改善、予約システム連携 | 初期3〜8万円 + 月3,000〜5,000円 |
| 12ヶ月以降 | 商品数増加、店頭外売上ニーズ、観光需要 | EC、多言語対応、再設計提案 | 案件別見積もり |

### 提案条件

- 1顧客に対する積極提案は原則 `四半期に1回まで`
- 提案は必ず `現状の課題` `提案内容` `期待効果` `概算費用` の4点でまとめる
- 「なんとなく売れそう」では提案しない。必ず数値か顧客発言を根拠にする
- 回収見込みが弱い施策は、先に無料の小改善を1つ試してから提案する

### 優先度の決め方

1. 既存顧客の困りごとが明確
2. 現在のサイトデータで改善余地が説明できる
3. 既存テンプレートを流用できる
4. 顧客が返信しやすい形で提案できる

この4条件を満たす提案だけを優先する。テンプレート化できない提案は、受注できても将来のボトルネックになりやすい。

## 解約防止

解約防止は、値引きで引き留めることではなく、顧客が「今やめる合理性」を感じる前に価値を再定義することだ。

### 早期警戒シグナル

| シグナル | 危険度 | 初動 |
|---|---|---|
| 2ヶ月連続でレポート未返信 | 中 | 次回レポートで質問を1つ入れる |
| 3ヶ月連続で問い合わせゼロ | 中 | 導線か集客のどちらを先に改善するか提案する |
| 「費用対効果が分からない」と言われる | 高 | 年間成果の可視化を即作る |
| 6ヶ月以上更新依頼がない | 中 | 活用提案かダウングレード相談を出す |
| 請求遅延が7日超 | 高 | 支払い確認と契約継続意思を確認する |

### 顧客ヘルス判定

簡易ヘルススコアで十分である。複雑な採点は続かない。

| 項目 | 配点 |
|---|---:|
| 月次レポートへの反応が60日以内にある | 20 |
| 請求遅延がない | 20 |
| 直近90日で流入または成果が大幅悪化していない | 20 |
| 直近6ヶ月で更新または改善の接点がある | 20 |
| 費用に対する不満が明示されていない | 20 |

- 80〜100点: 健全。通常運用
- 50〜79点: 要注意。次回レポートで相談導線を作る
- 0〜49点: 解約リスク高。電話か面談を提案する

### 解約防止の具体策

- 毎月のレポートに `累計訪問者数` `累計問い合わせ数` `実施した保守内容` のいずれかを入れる
- 年1回は年間サマリーを送り、単月では見えにくい価値を見せる
- 解約予兆がある顧客には、値下げより先に `ダウングレード` `改善スプリント` `一時休止` の順で選択肢を出す
- 解約の手順は明確にしておく  
  解約しづらさは一時的な延命にはなるが、紹介と再契約の機会を失いやすい

## KPIダッシュボード

ダッシュボードは1画面で見切れる構成にする。複雑なBIより、毎月更新できることを優先する。

### 追うべきKPI

| KPI | 定義 | 目標 | 警戒ライン | 更新頻度 |
|---|---|---:|---:|---|
| Active Clients | 保守契約中の顧客数 | 15社 | 25社超 | 月次 |
| MRR | 月額保守売上合計 | 12万円 | 前月比マイナス2ヶ月 | 月次 |
| Average MRR per Client | `MRR / 契約社数` | 8,000円以上 | 6,000円未満 | 月次 |
| Gross Churn | 解約MRR / 前月MRR | 3%未満 | 5%以上 | 月次 |
| Net MRR Expansion | 既存顧客の増額差分込みMRR成長 | プラス維持 | 2ヶ月連続マイナス | 月次 |
| Manual Hours | 保守に使った総手動時間 | 15時間以内 | 20時間超 | 月次 |
| Report On-time Rate | 期限内送付率 | 95%以上 | 90%未満 | 月次 |
| P1 MTTR | 障害復旧までの平均時間 | 4時間以内 | 8時間超 | 月次 |
| Upsell Pipeline | 提案済み案件の見込み額 | MRRの1ヶ月分以上 | 0が2ヶ月継続 | 月次 |
| Health Score Red Count | ヘルススコア49点以下の顧客数 | 0〜1社 | 2社以上 | 月次 |

### キャパシティ管理ルール

保守社数は単純件数ではなく、標準換算で見る。

- ライト = 0.5社分
- スタンダード = 1.0社分
- プレミアム = 1.5社分

標準換算で `20社分` を超えたら、以下のいずれかを実施する。

1. 新規保守の受注条件を厳しくする
2. 低単価顧客のプラン見直しを進める
3. 毎月の手動作業を減らす改善に1週間使う

## リスク登録簿

| リスク | 早期兆候 | 影響 | 予防策 | 発生時対応 |
|---|---|---|---|---|
| 自動更新で本番不具合が出る | Dependabot後のPreview差分 | 顧客サイト毀損 | パッチ/マイナーでもCIとPreview確認 | 自動化停止、直前安定版へ戻す |
| 監視通知の見落とし | 通知過多、誤検知増加 | 障害対応遅延 | 通知先を二重化、月次で閾値見直し | 監視ルールを絞り再設定 |
| ドメイン更新漏れ | 台帳未更新、請求失敗 | サイト停止 | 60日前と30日前通知、自動更新 | 緊急更新と顧客連絡を同時実施 |
| スコープ肥大化 | 軽微修正が常態化 | 利益率低下 | 依頼分類ルールを先に共有 | 上位プランか別見積もりへ切り替え |
| レポートが形骸化する | 返信ゼロ、送付遅延 | 解約増加 | 1通1提案、短文化、台帳記録 | 年間サマリーで価値を再整理 |
| 請求遅延が増える | 未入金7日超 | キャッシュ悪化 | 自動請求、失敗通知 | 継続意思確認、条件見直し |
| 体調不良や不在で止まる | 1人依存 | 全業務停止 | 手順書、復旧手順、優先顧客整理 | P1のみ対応に絞り通常対応を延期 |
| 顧客成果が弱く不満が出る | 問い合わせ減少、費用対効果の質問 | 解約・値下げ圧力 | 早期に改善仮説を出す | 改善スプリントかダウングレード提案 |

リスク登録簿は作って終わりではなく、解約や障害が起きた月に更新する。問題が起きたのに表が変わらないなら、登録簿は機能していない。

## 30/60/90日ローンチ計画

### 最初の30日: 基盤をそろえる

目標は「案件ごとに違う保守」をやめること。

- 全顧客の保守台帳を1つに集約する
- プラン別の対応範囲、返信目安、軽微修正の定義を固定する
- 監視、GA4、請求、ドメイン更新日の有無を棚卸しする
- 月次レポートのテンプレートを作る
- 3社だけ先行して Dependabot と監視フローを整備する

30日時点の到達条件:

- 保守中顧客の台帳が100%埋まっている
- P1/P2/P3 の分類ルールが自分の中で固まっている
- 月次レポートの初版テンプレートが完成している

### 31〜60日: 自動化と定例化を始める

目標は「忘れない運用」に変えること。

- Dependabot とデプロイ確認の週次ルーチンを固定する
- 監視通知、請求失敗通知、ドメイン期限通知を整える
- 月次レポート草案の自動生成を導入する
- 木曜の軽微修正バッチを試す
- 作業時間の記録を開始し、採算が合わない顧客を洗い出す

60日時点の到達条件:

- 月次レポートを1サイクル遅延なく回せている
- 依存関係更新の処理ルールが運用に乗っている
- 保守内と保守外の線引きで迷うケースが減っている

### 61〜90日: 成長導線を乗せる

目標は「守るだけの保守」から「伸ばす保守」に変えること。

- ヘルススコアを台帳へ追加する
- アップセルトリガーに該当する顧客を抽出する
- 四半期提案テンプレートを用意する
- 年間サマリーのひな型を作る
- KPIダッシュボードを見て、値上げ候補・ダウングレード候補を整理する

90日時点の到達条件:

- 四半期アップセル提案を少なくとも1回実施している
- 解約リスク顧客に対する予防アクションを実行している
- 手動時間とMRRのバランスで次の打ち手を決められる状態になっている

## Ready Checklist

Phase 6 の運用開始前に満たすべきチェックリスト。

- [ ] 保守契約の有無とプランが顧客ごとに明確
- [ ] 連絡窓口が1つに統一されている
- [ ] GitHub / 本番環境 / ドメイン管理情報が台帳化されている
- [ ] Uptime監視が有効
- [ ] アクセス計測が有効、または初月レポート代替案がある
- [ ] 軽微修正の定義と回数制限がある
- [ ] P1/P2/P3 の判定ルールがある
- [ ] 月次レポートの送付先担当者が決まっている
- [ ] 請求方法と未入金時の対応ルールがある
- [ ] 解約時の引き継ぎ方針が決まっている

## Done Checklist

このフェーズが「運用に乗った」と言える状態のチェックリスト。

- [ ] 月次レポートを2ヶ月連続で期限内送付できている
- [ ] Dependabot と監視フローが週次運用として定着している
- [ ] 作業時間を顧客別に記録し、採算判断に使えている
- [ ] 保守内/保守外の判定を迷わず返せる
- [ ] アップセルトリガーに基づく提案を少なくとも1件送っている
- [ ] ヘルススコアで要注意顧客を把握できている
- [ ] 解約防止アクションをテンプレート化できている
- [ ] KPIダッシュボードを毎月更新できている
- [ ] 標準換算社数に対して手動工数が許容範囲内に収まっている
- [ ] 顧客ごとの特例ではなく、共通ルールで保守を回せている

---

このフェーズの完成形は、保守作業がなくなることではない。自分が毎回ゼロから判断しなくても、同じ品質で回せることが完成条件である。Phase 6 が整うほど、制作案件の受注を増やしても運営が崩れにくくなる。
