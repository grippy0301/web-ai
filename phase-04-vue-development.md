## **フェーズ概要**

Phase 4 の目的は、Phase 3 で確定した要件をもとに、**再利用テンプレートを使って短時間で本番品質の Vue サイトへ落とし込むこと**です。ここで重視するのは「凝った作り込み」ではなく、**速く、壊れにくく、次案件にも流用できる形で仕上げること**です。

このフェーズの成果物は、単なる見た目の完成ではありません。以下の3点が揃って初めて完了とみなします。

- ビルド可能な Vue 3 実装一式
- CMS または静的データの運用方針が固まった状態
- Phase 5 に渡せる品質証跡と引き継ぎ情報

運用原則は次の4つです。

- 新規デザインをゼロから起こさない
- コードより先にテンプレートとページ構成を決める
- 品質ゲートを通らない装飾は後回しにする
- 1案件で作った部品は、次案件で再利用できるかを常に判断する

## **成功指標**

| 指標 | 目標値 | 判断ルール |
| --- | --- | --- |
| 総開発時間 | ライト 5〜7時間 / 標準 8〜10時間 | 10時間を超える見込みが出た時点で追加要件かスコープ過大と判断する |
| 初回プレビュー到達 | 着手後 60分以内 | 60分で画面が出ない場合、依存追加を止めてテンプレート基盤を見直す |
| テンプレート再利用率 | 70%以上 | 新規実装は 30% 以内に抑える |
| Lighthouse Performance | 90以上 | 90未満なら画像・JS分割・依存削減を優先修正 |
| Lighthouse SEO | 100 | meta 未設定や構造化データ不足を残したまま次工程へ進めない |
| Lighthouse Accessibility | 95以上 | 95未満、または重大な axe エラーがある場合は差し戻し |
| 型安全性 | TypeScript エラー 0件 | `any` 追加で逃がさない |
| 重大不具合 | 0件 | ナビゲーション不能、フォーム不能、表示崩れは handoff 不可 |
| CMS更新確認 | 主要項目3件以上を更新して反映確認 | 反映確認ができない場合は CMS 導線を未完了扱いにする |

## **前提条件**

着手前に、以下の条件が最低限揃っている必要があります。

| 分類 | 必須項目 | 着手判断 |
| --- | --- | --- |
| 契約 | 契約締結済み / 着手金入金済み | 未完了なら着手しない |
| 要件 | Phase 3 の高優先度確認事項が 0 件 | 1 件でも残っていれば要件確定を優先 |
| 情報 | 店舗名 / 住所 / 営業時間 / 問い合わせ導線 / ページ構成が確定 | 欠けていれば仮実装止まりに留める |
| デザイン材料 | ブランドカラー / 参考サイト 1件以上 / 使ってよい写真素材 | Hero 画像が未確定なら公開用仕上げに進まない |
| 技術 | テンプレートリポジトリ / Node 実行環境 / デプロイ先候補が確定 | 基盤未整備なら Phase 4 を始めない |
| 運用 | CMS を使うか、静的運用にするかの方針が決定 | ここが曖昧なまま実装しない |

追加の判断ルール:

- 1ページLPで、更新頻度が低い案件は CMS 必須ではありません。
- 営業時間、メニュー、告知の更新を顧客が自走したい場合は CMS を採用します。
- フォーム送信先が決まっていない場合は、見た目だけ先行実装しても完了扱いにしません。

## **スコープと非目標**

### **スコープ**

Phase 4 で扱う範囲は以下です。

- Vue 3 + Vite + TypeScript ベースのフロントエンド実装
- レスポンシブな基本ページ構成の組み立て
- セクション単位の再利用コンポーネント適用
- 必要最小限の CMS 接続、または静的データ接続
- LocalBusiness を前提とした基本SEO実装
- フォーム、地図、レビュー、ギャラリーなどの基本導線整備
- Phase 5 に渡すための品質確認と引き継ぎ資料整備

### **非目標**

以下は標準Phase 4の対象外とします。

- 独自バックエンド開発
- 会員機能、ログイン、決済、予約台帳などの業務システム化
- 無制限のデザイン探索
- 過剰なモーション演出やブランド特化のフルスクラッチ表現
- 多言語対応の本格運用
- 納品後の運用体制まで含むSOP整備

判断ルール:

- 要件が「既存セクションの差し替え」で表現できない場合は、追加見積もり候補です。
- 新機能が外部SaaS連携や認証を伴う場合、標準実装から外して別タスク化します。
- 追加要望が「ページ追加」ではなく「業務フロー追加」に見える場合は、Phase 4 ではなく再要件定義に戻します。

## **実装ワークフロー**

ソロ運営では、作業順よりも**止める判断**が重要です。以下の順で進めると、手戻りを最小化できます。

| ステップ | 目的 | 目安時間 | 主な成果物 | 止める判断 |
| --- | --- | --- | --- | --- |
| 0. 着手判定 | 入力確認とスコープ固定 | 20〜30分 | 着手メモ / 欠品一覧 | 必須情報不足なら実装を始めない |
| 1. 基盤立ち上げ | テンプレート複製とブランド反映 | 30〜45分 | 起動済みプロジェクト / カラートークン | 45分で表示できないなら依存追加を凍結 |
| 2. 情報設計 | ルート、セクション順、データ接続方針を決める | 30〜45分 | サイトマップ / データ表 | ここで未決定項目が多ければ UI 実装に進まない |
| 3. ページ骨格実装 | Hero から CTA まで仮データで組む | 90〜150分 | Home もしくは主要ページ骨格 | 3時間時点で骨格未完成ならオプション機能を削る |
| 4. 実データ統合 | CMS または静的データ反映 | 60〜90分 | 表示反映 / ローディング・エラーUI | API 仕様で迷ったら local mock を固定して先へ進める |
| 5. 品質ゲート | SEO / 性能 / A11y / フォーム確認 | 60〜90分 | スコア確認 / 不具合修正 | 基準未達なら装飾ではなく原因から直す |
| 6. 引き継ぎ準備 | Phase 5 に渡す資料を整える | 20〜30分 | handoff メモ / 環境変数例 / 残課題 | 口頭前提のまま渡さない |

実務ルール:

- セクション生成順は `Hero → Services → About → Reviews → Gallery → Access → Contact → Footer` を基本とします。
- 1セクションにつき AI 生成の大きなやり直しは 2 回までです。
- 3回目の修正が必要なら、AI に再投げせず手で削って整えた方が速いと判断します。
- レスポンシブ調整は全セクション完成後ではなく、各セクション完成時点で最低限確認します。

## **テンプレート戦略**

Phase 4 では「どのテンプレートで戦うか」を最初に決めることが工数管理の核心です。

### **テンプレートの選び方**

| テンプレート | 想定案件 | 想定工数 | 標準構成 | 採用ルール |
| --- | --- | --- | --- | --- |
| T1: LP型 | 1ページ完結、問い合わせ中心 | 5〜7時間 | Hero / About / Services / Reviews / Access / Contact | 最優先。迷ったらまずこれ |
| T2: 標準複数ページ型 | 3〜5ページ、サービス説明あり | 8〜10時間 | T1 + 詳細ページ + ルーティング | ページ追加理由が明確なときのみ採用 |
| T3: 拡張型 | 6ページ以上、独自機能あり | 10時間超 | 個別設計が必要 | 標準Phase 4ではなく再見積もり候補 |

### **運用ルール**

- blank から作らず、必ず既存テンプレートを複製して始めます。
- 変更対象は原則として「色・書体・写真・セクション順・カード表現」に限定します。
- `BaseButton`、`BaseImage`、`SectionHeading` などのUIプリミティブは先に固定し、後から案件ごとに崩さないようにします。
- 1案件で新規に増やすセクション種別は **最大2つまで** に抑えます。
- 新セクションは、次案件でも再利用できる見込みがあるときだけ採用します。

追加の判断ルール:

- 1ページLPなら Vue Router を無理に入れません。アンカー遷移で十分ならその方が速くて壊れにくいです。
- Pinia は「複数セクションで共有する情報がある」「CMSキャッシュが必要」のどちらかを満たすときだけ使います。
- ギャラリー写真が弱い案件では、Gallery を削って About と CTA を強めた方が成果につながります。

## **CMS / データフロー**

データは「誰が更新するか」で設計します。更新しないものまで CMS に入れると、実装も運用も遅くなります。

### **基本フロー**

```
Phase 3 要件定義
  ↓
コンテンツ整理表作成
  ↓
AI で初稿生成 + 人手で事実確認
  ↓
microCMS または静的データへ投入
  ↓
Vue composable / store で取得
  ↓
各セクションへ描画
  ↓
更新テストと品質確認
  ↓
Phase 5 へ引き継ぎ
```

### **推奨データ分割**

| データ | 置き場所 | 理由 |
| --- | --- | --- |
| 店舗基本情報、営業時間、Hero文言 | `site-info` | 全体で使い回す単一情報だから |
| サービス一覧 | `services` | 追加・並び替え需要が出やすいから |
| ギャラリー | `gallery` | 画像差し替えが発生しやすいから |
| レビュー | `reviews` | 表示件数や更新頻度を管理しやすいから |
| 色、余白、レイアウトルール | コード側 | CMS に入れると崩れやすいから |
| 一時的なAB案や没データ | CMS に入れない | 公開運用データと混ざると事故になるから |

### **運用ルール**

- フロントエンドで使う API キーは**読み取り専用**に分けます。
- CMS の並び順は `createdAt` ではなく `displayOrder` で管理します。
- 画像には必ず `alt` を付け、空欄で公開しません。
- 主要情報が 1 画面に収まる案件では、CMS を増やしすぎず `site-info` 中心で始めます。
- 顧客が更新しない項目は、CMS ではなく静的データのままでも構いません。

判断ルール:

- 顧客更新が月1回未満で、変更箇所が営業時間と告知程度なら静的運用も選択肢です。
- 顧客が自分で更新したいと言っていても、更新対象が曖昧なら最初は `site-info` と `services` の2本に絞ります。
- 画像点数が多い案件だけ `gallery` を採用し、素材不足の案件では無理に作りません。

## **パフォーマンス / SEO / アクセシビリティゲート**

このフェーズでは、見た目が整っていても品質基準を通らなければ未完了です。

| 領域 | 合格基準 | 確認内容 | 未達時の優先対応 |
| --- | --- | --- | --- |
| Performance | Lighthouse 90以上 | LCP / CLS / INP / 初期JS / 画像容量 | Hero画像圧縮、依存削減、遅延読込、コード分割 |
| SEO | Lighthouse 100 | title, description, canonical, OGP, LocalBusiness, 見出し構造 | meta と構造化データを先に埋める |
| Accessibility | Lighthouse 95以上 + axe 重大0件 | キーボード操作、ラベル、コントラスト、focus、alt | 色とラベルと見出し順を最優先で修正 |
| Responsive | 375 / 768 / 1280 で破綻なし | Hero、カード、フォーム、フッター | コンポーネント単位で幅と余白を調整 |
| Form | 送信成功と失敗が確認できる | 必須入力、エラー表示、送信先、通知 | 送信先固定、文言整理、失敗時UI追加 |

### **合格のための実務ルール**

- Hero画像は最初から軽量化前提で扱い、後から圧縮する前提で巨大画像を置かない
- 画像の `width` / `height` は必ず持たせて CLS を防ぐ
- ページごとの `h1` は1つに限定し、見出し飛びを作らない
- フォームには `label`、必須表示、送信結果通知を必ず付ける
- モーションは `prefers-reduced-motion` を考慮し、装飾より可読性を優先する

失敗時の是正順:

1. 画像最適化
2. 不要ライブラリ削除
3. レイアウト崩れ修正
4. メタ情報と構造化データ補完
5. 色とフォーカスの見直し

## **タイムボックス規則**

ソロ運営では「どこまで作るか」より「どこで打ち切るか」が利益を守ります。

### **基本ルール**

1. 60分以内にプレビュー表示まで到達しなければ、追加ライブラリ導入を止めます。
2. 総工数の 50% を使った時点で、主要ページ骨格が見えていなければスコープを削ります。
3. 総工数の 70% を超えたら、新機能追加を止めて品質確認へ移行します。
4. 1つのUI課題に 20分以上詰まったら、別解に切り替えるかテンプレートへ戻します。
5. 1案件で新規依存を追加するのは原則1個までです。

### **削減の優先順位**

工数超過の兆候が出たら、以下の順で削ります。

1. 複雑なアニメーション
2. 低優先度のギャラリー演出
3. 詳細ページの装飾
4. CMS 対象の拡張
5. 非必須のコンポーネント新規作成

### **逆に削ってはいけないもの**

- 問い合わせ導線
- 営業時間、住所、地図などの基本情報
- モバイル表示
- SEO 基本設定
- アクセシビリティの重大エラー修正

## **リスクレジスター**

| リスク | 起きる場面 | 影響 | 予防策 / 対応 |
| --- | --- | --- | --- |
| 要件の再解釈が発生する | 実装中に「やはりこうしたい」が出る | 手戻り、工数超過 | Step 0 で凍結範囲を明文化し、追加要望は別管理にする |
| 素材不足 | Hero 画像やレビューが弱い | 仕上がり低下 | 先に素材棚卸しし、不足時は代替案を即決する |
| AI生成コードが冗長 | Cursor 生成をそのまま採用する | 保守性低下、性能悪化 | 生成後に削る前提で見る。3回目以降は手修正 |
| CMS 設計が重い | 全項目を CMS 化しようとする | 初期工数増大 | 顧客が更新する項目だけを CMS に置く |
| API キー管理ミス | フロントに強い権限キーを置く | セキュリティ事故 | 読み取り専用キーを分離し、書き込み権限を出さない |
| パフォーマンス劣化 | 大画像、重い依存、過剰演出 | 品質ゲート未達 | Hero 画像圧縮、依存最小化、後回し判断を徹底する |
| フォーム不達 | 送信先設定や迷惑メール問題 | 問い合わせ損失 | 送信成功、失敗、通知先までテストする |
| スコープクリープ | 営業時の口約束が混ざる | 利益圧迫 | Phase 3 文書との差分を追加見積もり候補として管理する |
| 顧客確認の遅れ | 文言や写真確認が止まる | Phase 5 以降も遅延 | 確認待ち項目を分離し、実装可能部分だけ先に詰める |

## **Phase 5 への引き継ぎ**

Phase 5 は「公開しても事故らない状態」に持っていく工程です。Phase 4 からは、見た目の完成品ではなく、**公開準備に必要な判断材料一式**を渡します。

### **引き継ぎ内容**

| 項目 | 内容 | 必須 |
| --- | --- | --- |
| 実装一式 | Vue プロジェクト、ビルド可能状態 | 必須 |
| 環境変数例 | `.env.example` 相当の一覧 | 必須 |
| CMS 構成メモ | どの項目をどこで更新するか | CMS採用時は必須 |
| QA 証跡 | Lighthouse / axe / 画面確認メモ | 必須 |
| フォーム設定 | 送信先、通知先、確認済み条件 | 必須 |
| 既知の制約 | 未対応事項、保留事項、追加見積もり候補 | 必須 |
| 公開前注意点 | ドメイン、OGP、画像差し替え、最終確認事項 | 必須 |

### **handoff の判断ルール**

- `npm run build` 相当のビルド通過を前提とする
- 型エラーが残っている状態では渡さない
- 既知の未解決事項は「口頭」ではなく文書で残す
- Phase 5 でしか直せない内容と、Phase 4 で直すべき内容を分けて記録する

Phase 4 側で解消しておくべきもの:

- レイアウト崩れ
- コンテンツ欠落
- フォームUI不備
- SEOタグ未設定
- アクセシビリティ重大エラー

Phase 5 に引き渡してよいもの:

- 本番ドメイン差し替え
- DNS / CDN / キャッシュ設定
- 最終公開判定
- 納品物整理と顧客向け操作説明

## **30 / 60 / 90日ロールアウト**

このフェーズは1案件ごとに回すだけでなく、**3か月で標準化して利益率を上げる**前提で運用します。

| 期間 | 重点テーマ | 実施内容 | 到達目標 |
| --- | --- | --- | --- |
| 0〜30日 | 基盤固定 | テンプレート1本化、チェックリスト整備、主要セクション安定化 | 「毎回違う作り方」を止める |
| 31〜60日 | 再利用強化 | 業種別バリエーション追加、CMS投入手順の簡略化、AIプロンプト整理 | 2案件目以降の立ち上がりを速くする |
| 61〜90日 | 運用最適化 | QA証跡テンプレート化、工数ログ分析、削る判断の標準化 | 平均開発時間と差し戻し率を下げる |

### **30日目までにやること**

- ベーステンプレートを1本に絞る
- Hero / Services / About / Access / Contact を安定部品化する
- Ready / Done チェックを毎案件で使う

### **60日目までにやること**

- 上位業種向けの差し替えパターンを 2〜3 種類用意する
- `site-info` と `services` の運用を標準化する
- AI に投げるプロンプトを部品化し、毎回ゼロから書かない

### **90日目までにやること**

- 案件別の実工数を比較し、時間超過の原因を潰す
- Phase 5 への引き継ぎ資料を定型化する
- 「どこで削るか」の判断をテンプレート化し、利益が出る作り方を固定する
